Ownership 키 생성
— 스펙 문장 하나에서 키가 어떻게 만들어지나

사람이 뽑든 LLM이 뽑든 같은 문자열 — 추출·정규화·검증의 결정 절차 · 2026-07-03

30초 요약
문제
dedup 게이트는 “한 키 = 한 스펙”을 비교하지만, 그 키를 스펙 문장에서 어떻게 뽑아 만드는지는 사람·LLM 판단에 맡겨져 있었다 — 결정성이 반쪽.
해법
스펙 문장 → ①추출(어느 단어가 키가 되나) → ②정규화(고정 규칙) → 같은 문자열. 규칙이 게이트 코드에 박혀 있어 사람 = LLM 동일 결과.
당신이 할 일
FR 문장에서 소유하는 명사=Entity · 동사=Capability verb · route=Surface를 이 절차대로 뽑아 ## Ownership에 적는다.

01왜 필요한가 — 키의 결정성이 반쪽이었다

키의 생애는 세 관문을 지난다. 마지막 비교는 이미 결정적인데, 앞의 추출·정규화가 판단에 맡겨져 있으면 같은 개념이 서로 다른 키로 새어 든다.

스펙 문장 / 코드  ──①추출──▶  키 후보  ──②정규화──▶  표준 키  ──③비교──▶  충돌?
                 (규칙 없음)           (공백·소문자만)       (결정적 ✅ 기존 게이트)
                    ↑ 여기가 비어 있었다      ↑ 여기를 규칙으로 강화
막지 않으면 생기는 일

같은 “추천” 개념을 사람 A는 Recommendation, 사람 B는 recommendation, LLM은 post /api/Recommend/로 적는다. dedup은 같은 것을 다른 키로 봐서 중복을 놓친다(거짓음성). 반대로 참조일 뿐인 걸 소유로 적으면 거짓충돌(거짓양성). → 그래서 추출·정규화를 절대 규칙으로 박는다.

02키 3종 — 무엇을 스펙이 “소유”하나

게이트가 다루는 소유 키는 세 종류다. 각각 스펙 문장의 어느 자리에서 나오는지가 정해져 있다.

Entity

도메인 객체 · 테이블

시스템이 생성·소유하는 데이터. 스키마 테이블/타입명이 진실(SSOT).

recommendation
pjt_projects
Surface

route · 화면 · job

외부에 노출되는 표면. API 엔드포인트·화면 경로·이벤트/배치.

POST /api/recommend
event:project.created
Capability

entity . verb

“무엇을 할 수 있나”. 대상 Entity와 동작의 조합.

recommendation.create
staff.assign

03스펙 문장 하나 → 키 (전 과정)

모르는 사람이 봐도 따라오도록, 실제 FR 문장 하나에서 키를 뽑아 만드는 전 과정을 보인다.

출발점 — 스펙 문장 (FR · EARS · 정본 영어)

FR-002 (event): WHEN a manager requests staff recommendations for a project, THE SYSTEM SHALL create a recommendation.
소유 명사 → Entity 동사 → verb 읽기만 → Dependency

①추출 — 문장의 어느 자리를 키로 가져오나

키 종류문장에서 뽑는 자리 (규칙)이 문장에서
EntitySHALL이 생성·소유하는 명사 = aggregate root. 스키마 테이블/타입명이 SSOTrecommendation
Capability그 Entity + 핵심 동사 1개 → entity.verb (동사를 허용 verb 집합에 매핑)recommendation + create
Surface그 FR을 노출하는 route 정의(코드가 SSOT — 문장이 아니라 엔드포인트에서)POST /api/recommend
Dependency
(참조)		SHALL이 읽기만 하는 명사(소유 아님) → dedup 제외staff, project

핵심 구분: 시스템이 만들어내는 명사만 소유(Entity), 읽어 쓰는 명사는 참조(Dependency). 이 한 줄이 dedup 거짓양성을 없앤다.

②정규화 — 뽑은 후보가 최종 문자열이 되는 과정 (저자가 달라도 동일)

저자 A와 저자 B(또는 LLM)가 같은 문장에서 조금씩 다르게 뽑아도, 게이트 normalizeKey를 통과하면 같은 문자열로 수렴한다. 아래는 실제 함수를 돌린 출력이다.

저자 A                        저자 B                        → 게이트 normalizeKey 출력      [검증]
[Entities]     "recommendation"       "Recommendation"       → "recommendation"          ✓ 동일
[Capabilities] "recommendation.create" "Recommendation.Create" → "recommendation.create"   ✓ 동일
[Surfaces]     "POST /api/recommend"   "post /api/Recommend/"  → "POST /api/recommend"     ✓ 동일
──────────────────────────────────────────────────────────────────────────
✅ 서로 다른 입력 → 고유 출력 1개. 사람 = LLM 최종 키 문자열이 완전히 같다.

결과 — 이 문장 하나가 만들어내는 블록

## Ownership
- **Entities**: recommendation
- **Surfaces**: POST /api/recommend
- **Capabilities**: recommendation.create

## Dependencies          # SHALL이 읽기만 하는 명사 → dedup 제외
- **Entities**: staff, project

이제 check-ownership 게이트가 이 키들을 전 스펙과 대조해 “한 키 = 한 스펙”을 CI에서 강제한다.

04정규화 절대 규칙 (결정성의 심장)

일반 규칙은 게이트 코드에 고정(모든 프로젝트 동일), 도메인 가변부(verb 집합·path param 표기)만 sdd.config.json.

카테고리출처 (SSOT)정규화 규칙 (결정적)예
Entity스키마·모델·마이그레이션의 테이블/타입명식별자 그대로 + trim().toLowerCase(). 단복수 임의변환 ✗(스키마가 진실)recommendation
Surfaceroute·이벤트·job 정의<METHOD> <path> — METHOD 대문자, path 소문자, param {name} 표준형(:id·<id>→{id}), trailing slash 제거. 이벤트=event:<name>, job=job:<name>POST /api/recommend/{id}
Capability공개 함수·핸들러 + Entity<entity>.<verb> — 점 정확히 1개, 소문자, verb ∈ 허용집합recommendation.create

Surface 규칙은 특히 강력하다 — 대소문자·파라미터 문법(:id/<id>/{id})·trailing slash가 제각각이어도 하나로 수렴한다(실측):

[Surfaces] "get /api/Staff/:id"    → "GET /api/staff/{id}"
[Surfaces] "GET /api/staff/{id}/"  → "GET /api/staff/{id}"
[Surfaces] "Get /API/staff/<id>"   → "GET /api/staff/{id}"

05멈춤(STOP) 가드 — 수렴 못 할 애매함은 통과 못 한다

수렴시킬 수 있는 것(대소문자·표기)은 자동으로 수렴시키고, 수렴 못 할 애매함(유사어·형식 위반)은 validateKey가 통과 자체를 막는다. 그래서 사람·LLM이 임의로 갈라진 키를 만들 길이 없다(실측).

[Capabilities] "staff.suggest"   → ✗ STOP  미등록 verb "suggest" (recommend의 유사어)
[Capabilities] "staff.recommend" → ✗ STOP  미등록 verb "recommend" — capabilityVerbs 등록[리뷰] 후 진행
[Surfaces]     "recommend/{id}"  → ✗ STOP  메서드 없음 — "<METHOD> <path>" 형식 위반
임의 동의어 우회 금지

recommend·suggest 같은 변종이 키를 분산시키지 않도록, 도메인 verb는 capabilityVerbs에 등록해야만 통과한다(CRUD 기본 create/read/update/delete/list은 항상 허용). 등록은 specIdPrefixes와 같은 config 리뷰 관문 — 새 verb를 몰래 늘리는 대신 리뷰에 노출한다.

06왜 “사람 = LLM 동일”이 보장되나

결정성의 근거

추출이라는 창의적 행위는 사람·LLM이 하지만, 규칙과 검증은 게이트 코드에 고정돼 있고 언어중립이다. 그래서 누가·언제·어떤 도구로 뽑든 정규화를 통과한 최종 문자열은 동일하다. 이건 확률적 모델의 “아마 같음”이 아니라 함수의 결정성 — 그래서 CI에서 exit 1로 강제할 수 있다.
