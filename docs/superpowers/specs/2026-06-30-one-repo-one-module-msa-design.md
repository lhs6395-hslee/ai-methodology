# 설계 — 1 레포 = 1 모듈 + MSA 합성 (방법론 재정의)

> Status: **Draft (검토 대기)** · Date: 2026-06-30 · Scope: 방법론 문서 재기준화 + MSA 계약 프로파일 신설
> 관련: [STRUCTURE.md](../../../STRUCTURE.md) · [STORAGE.md](../../../STORAGE.md) · [APPLYING.md](../../../APPLYING.md) · [METHODOLOGY.md](../../../METHODOLOGY.md) · [DEDUP.md](../../../DEDUP.md) · [principles.md](../../../principles.md)

## 0. 한 줄 요약

SDD의 단위를 **1 레포 = 1 모듈**로 고정한다. 큰 프로그램은 **여러 모듈-레포의 MSA 합성**으로 구현하며, 다중 모듈일 때만 켜는 **선택형 "MSA 계약 프로파일"** 로 모듈 간 계약을 1급 SSOT로 강제한다. 강제는 *cross-repo 메가 게이트*가 아니라 **계약-앵커드 양방향 in-repo 검증 + 얇은 통합 레포(조립 명세)** 로 한다.

## 1. 배경 — 무엇이 문제였나

현 방법론은 **한 레포 안에 여러 모듈**을 가정한다:
- [STRUCTURE.md](../../../STRUCTURE.md): "큰 틀 = 모듈, 모듈 = 제품당 보통 10여 개, `MODULE_MAP.md`로 인덱싱."
- [APPLYING.md](../../../APPLYING.md) §2: "모듈을 10여 개로 정의, `MODULE_MAP.md`에 등록."
- [STORAGE.md](../../../STORAGE.md) §1: `sdd/MODULE_MAP.md` = 다중 모듈→spec 레지스트리.

이 가정은 (a) "한 레포가 곧 한 배포 단위"인 MSA 현실과 어긋나고, (b) 시스템이 커질 때 모듈 간 경계·계약을 규정할 자리가 없다. 모듈을 "독립"으로만 두고 사이를 규정하지 않으면 결과는 **분산 모놀리스**(서로 몰래 결합돼 같이만 배포되는 MSA의 실패형)다.

## 2. 결정 (확정)

1. **1 레포 = 1 모듈** (무조건). 한 레포 = 정확히 하나의 모듈(bounded context), 그 모듈은 **여러 spec**을 가진다.
2. 큰 프로그램 = **여러 모듈-레포의 MSA 합성**. 각 모듈-레포는 자기 `sdd/` + `sdd.config.json` + 게이트를 **독립적으로** 갖는다.
3. **MSA 계약 프로파일** = 다중 모듈일 때만 켜는 **선택 계층**. 모듈의 공개 인터페이스를 기계가독 계약으로 1급화하고 양방향으로 검증한다. 단일 모듈 레포는 비용 0.
4. **계약 SSOT = 생산(provider) 모듈 레포**가 소유. **통합 레포는 버전 핀·상호작용 규약만 참조**(얇게). 계약 원본을 통합 레포로 옮기지 않는다(중앙 결합 방지).

## 3. 새 모델

```
[ 큰 프로그램 = MSA 시스템 ]
   ├─ repo A = 모듈 A   (sdd/ + config + gate · spec 다수 · 공개 계약 owns)
   ├─ repo B = 모듈 B   (sdd/ + config + gate · spec 다수 · 공개 계약 owns)
   └─ repo C = 모듈 C   (sdd/ + config + gate · spec 다수 · 공개 계약 owns)
          │                    │                    │
          └─── 공개 계약(API/이벤트, 기계가독 schema)으로만 결합 ───┘
                              │
      [ 통합 레포 = 시스템 SSOT (얇음) ]
        - 어떤 모듈-레포들이 시스템을 구성하나
        - 각 모듈 공개 계약의 **버전 핀**(호환 매트릭스)
        - 상호작용 규약(동기 req/resp vs 비동기 event) — 벤더 중립
        ※ 계약 원본·모듈 코드·빌드 파이프라인은 소유하지 않음

모듈 내부(레포 안): spec ⇄ test(@covers) ⇄ code 사슬이 레포 안에서 닫힌다.
모듈 경계(레포 간): contract ⇄ provider/consumer 사슬이 계약 버전에 앵커된다.
```

**통합 레포의 무게가 곧 MSA 건전성:**

| 통합 레포 역할 | MSA에 | 이유 |
|---|---|---|
| 계약 **원본**을 중앙 소유 | ✗ 깎아먹음 | 모든 모듈이 한 곳에 의존 → 중앙 결합 부활 |
| 모듈 **코드**를 끌어모음 | ✗ 깎아먹음 | "1 레포=1 모듈" 위반, 모놀리스 회귀 |
| 모든 모듈을 한 파이프라인으로 빌드·배포 강제 | ✗ 깎아먹음 | 독립 배포가 깨짐 (분산 모놀리스) |
| 버전 핀·상호작용 규약만 **참조** | ✓ 강화함 | 소유는 모듈에, 통합 레포는 "조립 명세"만 |

## 4. Phase 1 — 방법론 재기준화 (1 레포 = 1 모듈)

문서·메타만 바뀌는 기반 작업. **먼저 한다.**

| 파일 | 지금 | 바뀔 방향 |
|---|---|---|
| [STRUCTURE.md](../../../STRUCTURE.md) | "큰 틀=모듈, 제품당 10여 개, MODULE_MAP로 인덱싱" | **1 레포=1 모듈**, 모듈 안에 spec 다수. 다중 모듈=다중 레포=MSA. "과편화 금지"는 *모듈 안 spec 과편화*로 의미 이동 |
| `MODULE_MAP.md` (역할) | 한 레포 안 **여러 모듈→spec** 레지스트리 | 이 레포 **단일 모듈 매니페스트**: 모듈 정체성(이름·bounded context) + 공개 계약(Surface) 포인터 + 그 모듈 spec 인덱스. 파일명은 유지(툴링 churn 최소화), 내용 재정의 |

Phase 1만으로도 방법론은 일관되게 작동한다(MSA 프로파일 없이 단일 모듈 레포로).

## 5. Phase 2 — MSA 계약 프로파일 (선택 계층)

다중 모듈 시스템에서만 켠다. 3요소.

**요소 1 — 공개 계약 = 기계가독 산출물, spec이 소유.**
모듈의 대외 인터페이스(공개 route·RPC·발행 이벤트)는 OpenAPI/proto/AsyncAPI 등 **스키마/IDL**로 표현하고, 그 계약을 그 모듈 spec의 `## Ownership > Surface`가 소유한다. 신설 아님 — [STRUCTURE.md](../../../STRUCTURE.md) "SSOT 3계층"의 **구조 SSOT 티어에 이미 "API=스키마/IDL/proto"** 가 있다. 공개 Surface는 그 계약 산출물을 *필수*로 끌어올린다. 계약 파일은 코드/구조라 `sdd/`가 아니라 정상 소스 트리(언어 관례 위치)에 두고, Surface가 그 경로를 참조한다.

**요소 2 — 양방향 in-repo 계약 테스트 (계약 버전에 앵커).**
- **Provider 쪽:** "내 코드가 내가 게시한 계약을 여전히 만족하는가" — in-repo 게이트, 강제 가능.
- **Consumer 쪽:** 의존하는 외부 모듈의 계약을 **로컬에 버전 핀(lock)** 으로 고정하고, "나는 그 핀된 계약 안에서만 소비하는가"를 in-repo로 검증. (lockfile 모델 — cross-repo CI 불필요.)
- 이것이 기존 `FR↔test↔code` 사슬에 **`contract↔provider/consumer`** 경계를 형제로 추가한다.

**요소 3 — 통합 레포 = 시스템 SSOT (얇음).**
- `SYSTEM_MAP.md` — 구성 모듈-레포 목록 + 각 공개 계약 위치 + **호환 버전 매트릭스**(핀).
- **상호작용 규약** — 동기(req/resp) vs 비동기(event) 중 무엇인지. ⚠ [principles.md §10](../../../principles.md) 벤더 중립 적용: **"이벤트 로그로 비동기" O / "Kafka" X.** 제품이 아니라 상호작용 *의미*만.
- (선택) 통합 레포 자신도 SDD-거버넌스 가능: 그 specs = 시스템/통합 레벨 FR.

## 6. 강제 메커니즘 — 무엇이 in-repo로 가능한가

| 검증 | 어디서 | 가능? | 방법 |
|---|---|---|---|
| 공개 Surface ↔ 계약 산출물 존재·소유 | provider 레포 | ✓ | 기존 ownership 게이트 확장(`check-contract`) |
| 코드 ↔ 게시 계약 일치 | provider 레포 | ✓ | provider 계약 테스트(언어 도구) |
| 소비 ↔ 핀된 계약 | consumer 레포 | ✓ | 로컬 계약 lock + consumer 계약 테스트 |
| 버전 매트릭스 내부 정합 | 통합 레포 | ✓ | SYSTEM_MAP 핀 일관성 체크(얇은 게이트) |
| 전 그래프 자동 호환성·breaking-change 전파 | — | ✗(보류) | 필요 증명 시 별도 |

핵심: **한 레포의 게이트가 다른 레포를 직접 강제하지 않는다.** 양쪽이 각자 CI에서 *공유 계약(핀)* 에 대해 독립 검증한다(consumer-driven contract 원리). 그래서 cross-repo CI 없이 강제가 성립한다.

## 7. YAGNI 경계 (지금 넣지 않는 것)

- 단일 오케스트레이션 파이프라인(모든 모듈 한 번에 빌드) — MSA 안티패턴, 넣지 않음.
- 전체 의존 그래프 자동 호환성·breaking-change 탐지 — 필요 증명 후.
- 계약 브로커(Pact 류) 도입 강제 — 원리만 채택, 제품 비강제(벤더 중립).
- MSA 프로파일은 **다중 모듈일 때만** 켠다 — 단일 모듈 레포는 요소 1~3 비용 0.

## 8. 범위·분해

- **Phase 1(재기준화)** 와 **Phase 2(MSA 프로파일)** 는 **각각 별도 구현 계획**으로 간다. Phase 1이 선행(기반), Phase 2가 후행(선택 계층).
- Phase 2의 게이트(`check-contract` 등)는 기존 4-런타임 게이트와 동일하게 출력 동일·config 구동 원칙을 따른다([principles.md §10](../../../principles.md)).

## 9. 미해결 질문 / 리스크

1. `MODULE_MAP.md` 파일명 유지 vs `MODULE.md`로 개명 — 본 설계는 **유지+재정의** 권장(툴링 churn 최소). 개명 원하면 `sdd-init.sh`·템플릿 동반 수정.
2. 계약 lock의 구체 형식(파일·해시·semver 범위) — Phase 2 구현 계획에서 확정.
3. 통합 레포를 키트가 스캐폴드할지(예: `sdd-init.sh --system`) — Phase 2 범위, 본 설계는 개념만.
4. 기존 단일-레포-다중-모듈로 적용한 프로젝트가 있다면 마이그레이션 지침 필요 — 현재 채택 프로젝트 유무 확인 필요(없으면 비고려).
