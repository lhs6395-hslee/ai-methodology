# 하네스 (Gap 3) — 비전·요구 캡처 (설계 전, 규칙 뒤)

> Status: **요구 캡처(설계 전)** · Date: 2026-06-30
> 순서: 규칙(Gap 1 = 1 레포 1 모듈 / Gap 2 = spec 입도)을 **먼저 굳히고**, 그 위에 하네스를 설계·구현한다. 이 문서는 *요구를 잊지 않으려* 모아둔 것 — 정식 설계는 규칙 완료 후.

## 목적

방법론의 **0~8 루프 + converge + dedup**을 **사람-개입(human-in-the-loop) 인터랙티브 하네스**로 오케스트레이션한다. 핵심은 *결정 지점마다 사람에게 의사를 묻고* 단계를 잇는 것. 판정/검증 로직은 대부분 **이미 게이트로 존재**하므로, 하네스 = **트리거 + 확인 게이트 + 단계 연결**(새 판정 로직을 처음부터 짜는 게 아님).

## 형태 (합의)

**(C) 포터블 플로우 계약 + 에이전트별 구현.** 상태머신·결정지점·프롬프트를 한 번 정의(계약), Claude Code를 1차 레퍼런스 구현으로. 게이트 4-런타임 패턴과 동일한 정신("동작은 정의, 구현은 런타임별") — [principles.md §10](../../../principles.md) 범용성 보존.

## 요구사항 (현재까지)

- **R1 spec→code:** spec 생성/변경 시 → 코드 검증 → 사람에게 "코드 생성/업데이트?" 의사 확인 → 생성 or 업데이트.
- **R2 code→spec:** 코드 변경 시 → 사람에게 "기존 spec 업데이트 vs 새 spec?" 의사 확인 → `/specify` update 또는 새 spec(+intent 입력) → `/analyze` → 사람 bless.
- **R3 spec 생성 완료 후 dedup:** 전 spec 분석 → 중복(구조적 `check-ownership` + 의미적 이웃 리뷰) + 입도(`check-spec-cohesion`) 확인 → 사람 확인 → 업데이트(통합/분할).
- **R4 (상시 sync) 파이프라인 후 spec↔code 일치 수시 감지:** 파이프라인이 돈 뒤에도 **항상/수시로** spec과 code가 sync인지 파악. drift 발견 시 표면화 → 사람 확인 → R1/R2로 흘림. ← *사용자가 추가한 요구.*

## 이미 있는 자산 ↔ 하네스가 더하는 것

| 요구 | 이미 있는 것 (탐지/판정) | 하네스가 더하는 것 |
|---|---|---|
| R1 | `check-fr-coverage` · `/converge`(top-down) | spec-저장 **트리거** + 생성/업데이트 확인 게이트 + TDD 구동 |
| R2 | `/converge`(bottom-up): 갭→intent→`/specify` update→bless | 코드-변경 **트리거** + 신규/개정 분기 질문 |
| R3 | `check-ownership`(구조) + 이웃 리뷰(의미) + `check-spec-cohesion`(입도) | spec 생성 직후 **자동 실행** + 확인 게이트 + 통합/분할 |
| **R4** | **`check-converge-drift`**(코드 변경·스펙 무변경) + **`check-orphan-surfaces`**(spec 없는 표면) + `check-fr-coverage` | **상시/스케줄 트리거**(커밋 훅·CI·watch·주기) + drift 사람 확인 게이트 + R1/R2 연결 |

→ **R4의 탐지는 이미 게이트 3종으로 존재.** 하네스는 그걸 *수시로 돌리고* drift를 사람에게 표면화하는 **상시 sync 루프** 층을 더한다.

## 미해결 (Gap 3 설계 단계에서 확정)

1. 플로우 계약 표기 형식 — 상태머신 스키마? 결정지점·프롬프트 명세 형식?
2. **트리거 메커니즘** — 커밋 훅 / CI / 파일 watch / 주기적. (R4 "수시"의 구체화: 어느 주기·어느 이벤트에 sync 점검?)
3. Claude Code 1차 구현 형태 — 스킬 + 슬래시명령 + `AskUserQuestion` + (선택)Workflow/훅.
4. 사람 확인 게이트 UX — 배치 확인 vs 단건 확인.
5. 하네스가 호출하는 게이트 집합·순서(파이프라인 정의)와 advisory↔strict 승격 시점.
