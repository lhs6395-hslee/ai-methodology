Spec-Driven Development · 범용 키트

"무엇·왜"와 "어떻게"를
항상 일치시키는 개발 방법

스펙(무엇을·왜)과 코드(어떻게)를 다른 서랍에 나눠 두고, 둘이 어긋나지 않도록 기계(게이트)가 강제하고 사람이 승인한다. Spec Kit · EARS · Superpowers를 한 골격으로 묶고, 언어·모델·인프라에 무관하게 같게 돈다.

비유 — 건축 도면과 건물. 도면(spec) 없이 건물(code)을 올리면 "왜 이렇게 지었지?"를 아무도 모른다. 도면만 그리고 건물과 안 맞으면 도면이 거짓말이 된다. 이 방법론은 도면과 건물을 묶어두고, 어긋나면 경보가 울리게 한다.

이 글은 하나의 가상 기능 — "프로젝트에 직원을 추천해주는 기능" — 을 처음부터 끝까지 따라가며 설명한다. 추천 서비스가 한 모듈, 그 안의 SPEC-001이 그 기능의 명세다.

출발점

대부분은 "바이브 코딩"에서 시작했다 — prompt-driven · feature 단위

AI 개발에서 가장 흔한 방식은 바이브 코딩이다 — 말로 기능을 던지면 코드가 나오고, "돌아가 보이면" 다음으로. 빠르고 · 무계획 · 직관적이라 지배적 방식이 됐다. 문제는 작업 단위인 "feature"가 너무 굵다는 것 — 겉은 ✓ 하나여도 속의 미완성이 숨는다.

왜 이 방식인가

바이브 코딩에서 명세 기반으로

왜 바이브 코딩이 대세였고, 왜 우리는 명세 기반으로 옮겼나 — 한눈에.

실제로 무엇이 달라지나 — 같은 일, 다른 방식

바이브 코딩이 방법론
작업 단위Feature (굵음)FR — 요구 하나하나
완료 기준"돌아가 보임"unit · smoke · deferred 회계
미구현안 보임 (기억에만)deferred로 명시
추적채팅 히스토리spec ↔ code ↔ test

쉽게 말하면 — 바이브 코딩은 "말하면 바로 만들기", 이 방법론은 "무엇을 만들지 먼저 적고, 빠짐없이 만들었는지 확인하기"다.

표에 나온 말 풀이 — 요구(FR)와 "회계"

기능 요청 하나를 요구(FR) 여러 개로 쪼개고, 각 FR을 아래 셋 중 하나로 표시한다. 이 분류가 "회계(accounting)" — 셋 중 아무것도 아니면 게이트가 통과시키지 않는다. 그래서 미완성이 숨지 못한다.

unit

자동 테스트로 검증됨

그 요구를 확인하는 단위 테스트가 있고, 코드와 @covers로 묶인다 — 기계가 매번 확인.

smoke

실행 증거로 검증됨

단위 테스트 대신 실제 동작 증거(CI 로그·스크립트·runbook)로 확인. 증거에 @verifies 태그를 남긴다.

deferred

아직 안 만듦 (미룸)

일부러 뒤로 미룬 요구. 이유를 적어 미완성임을 드러낸다 — 숨기지 않고 명시.

추적 — 한 요구가 spec ↔ code ↔ test에 묶인다

단점은 있다 — 그리고 이렇게 보완했다

명세 기반이 완벽하다는 말이 아니다. SDD가 흔히 지적받는 약점을, 우리는 다음 장치로 메운다.

SDD가 지적받는 단점우리의 보완
작은 수정에도 절차가 붙는 오버헤드Spec-Impact: none 탈출구·hotfix 경로로 사소한 변경은 가볍게 통과
코드와 스펙을 함께 관리하는 부담retag·smoke-scan이 연결을 자동 갱신 — 손품 최소화
스펙이 낡으면 오히려 오해를 부름drift 게이트가 어긋남을 상시 감지 — 낡기 전에 되돌림
회계를 형식적으로 채울 위험근거·담당(entityRegistry) 등록을 필수화 — 빈 양식 차단
"옳은 방향"까지 게이트가 보장하진 못함마지막 판단은 사람 승인 — 기계는 정합성, 사람은 옳음
탐색 단계엔 과할 수 있음탐색은 바이브로, 굳으면 채택하는 수명주기로 설계

언제 무엇을 쓰나 — 둘은 경쟁이 아니라 단계다

바이브 코딩이 맞을 때 (탐색 · 단기)이 방법론이 맞을 때 (운영 · 장기)
프로토타입 · MVP · 해커톤 · 데모오래 유지하고 계속 커질 코드
한 번 쓰고 버릴 코드 · 아이디어 검증프로덕션 · 핵심 시스템
요구가 아직 불확실 (탐색 중)요구가 굳음 · 정확성/규정이 중요
혼자 · 단기 작업여러 사람 · 여러 모듈 협업
"빨리 돌아가는 것"이 최우선"무너지지 않는 것"이 최우선

한 프로젝트 안에서도 섞인다 — 탐색은 바이브로 빠르게 훑고, 방향이 굳으면 그 부분부터 이 방법론으로 채택한다(채택 수명주기).

한 줄로: 탐색은 바이브 코딩으로 빠르게, 굳으면 명세 기반으로 붙든다. 단점을 없앴다고 말하지 않는다 — 장치로 메우고, "옳음"은 끝까지 사람이 승인한다.
왜 필요한가

AI는 코드를 잘 짠다 — 무너지는 건 그 주변이다

LLM은 구현엔 강하다. "직원 추천 함수 짜줘"라고 하면 잘 짠다. 문제는 그 코드를 둘러싼 것들이다. 한 달 뒤 "이 추천 로직 왜 이렇게 됐지?", "이거 이미 다른 데서 만들지 않았나?", "명세엔 A인데 코드는 B인데?" — 이 셋이 시간이 지나며 반드시 무너진다. 이 방법론은 그걸 구조 + 기계적 강제로 막는 것이 전부다.

① 흩어짐

요구사항이 채팅·머릿속·주석에 흩어져 단일 진실원(SSOT)이 없다. "무엇을 만들기로 했더라?"에 답할 곳이 없다.

② 표류(drift)

핫픽스로 코드만 고치고 명세는 그대로 둔다 → 시간이 지나면 도면과 건물이 따로 논다. 명세가 거짓말이 된다.

③ 중복

이미 "추천" 기능이 있는 줄 모르고 비슷한 걸 또 만든다. 진실이 두 곳으로 갈라진다.

무엇을 조합했나

4개의 기둥

혼자 만든 게 아니라, 검증된 도구 셋을 한 골격으로 묶고 거기에 "어디서나 같게 돈다"는 보장을 더했다. 앞 3개가 무엇을 하는가, 4번째가 어디서나 같게 돈다는 약속이다.

골격

GitHub Spec Kit

specify→plan→tasks 흐름 + 리뷰 명령. "스펙의 옳음"을 담당. 30개+ 에이전트 비종속.

요구 형식

EARS

요구를 모호하지 않은 5가지 문장틀로 강제. "한 요구 = 한 동작 = 검증가능".

구현·검증

Superpowers

TDD(테스트 먼저)·완료전검증·코드리뷰. "코드의 옳음"을 담당.

보장

범용성

언어·모델·DB·클라우드·CI 무엇이든 같게. 차이는 설정 한 장으로만 흡수.

왜 한 골격으로 묶나 — 셋은 서로의 구멍을 메운다

따로 쓰면 각자 한 부분만 잘하고 구멍이 남는다. 묶어야 무엇을(EARS로 정밀히) → 관리(Spec Kit) → 검증(Superpowers) 사슬이 닫힌다 — 하나라도 빠지면 끊긴다.

혼자면

Spec Kit 단독

spec을 관리하지만 FR이 모호해도 못 막고, 코드가 맞는지 검증하지 않는다.

혼자면

EARS 단독

요구를 검증 가능하게 만들지만 문장 형식일 뿐 — 관리도 검증도 못 한다.

혼자면

Superpowers 단독

코드를 검증하지만 "무엇이 명세인가·중복인가"를 모른다.

EARS 없으면 FR이 모호해 테스트로 못 묶고, Spec Kit 없으면 명세가 흩어지고, Superpowers 없으면 코드가 미검증이라 명세가 희망사항이 된다 — 그래서 한 골격. (+ 4번째 기둥 범용성이 어디서나 같게 돌린다.)

EARS 5패턴 — 우리 예제로 보면

"직원 추천"의 요구를 EARS로 쓰면 모호함이 사라진다. 예: WHEN 직원 조회 요청이 오면, THE SYSTEM SHALL 적합도 순으로 추천 목록을 반환한다.

패턴문장틀
UbiquitousTHE SYSTEM SHALL <응답> (항상)
EventWHEN <트리거>, THE SYSTEM SHALL <응답>
StateWHILE <상태>, THE SYSTEM SHALL …
UnwantedIF <비정상>, THEN THE SYSTEM SHALL …
OptionalWHERE <옵션>, THE SYSTEM SHALL …

금지: "적절히", "가능하면", should, 한 문장에 동작 2개, 측정 불가. — 모호하면 검증할 수 없고, 검증 못 하면 게이트가 못 잡는다.

⚠ EARS는 spec 전체가 아니라 FR 절에만 적용된다. 한 spec은 여러 절로 구성되고 절마다 형식이 다르다 — 그리고 코드와 @covers로 묶이는 추적 단위는 FR뿐이다. SC·NFR은 검증·측정의 기준이지 코드에 1:1로 매달리지 않는다.

분담: FR은 이 키트의 게이트(@covers·dedup·cohesion·completeness)가 추적·강제하고, 인수조건·SC는 Spec Kit 네이티브(/analyze·/checklist·/tasks)가 담당한다. check-spec-completeness는 SC·인수조건이 존재하는지만 advisory로 지킨다 — SC 충족은 런타임 지표라 빌드 게이트가 강제할 수 없다.

선택 절 Infrastructure Prerequisites의 "왜": 인프라 의존도 요구사항이라 spec에 적되 역량으로 적는다 — "내구성 있는 이벤트 로그 필요"(O) / "Kafka"(X). spec에 두는 이유는 그 의존이 추적되고, 배포 실제(IaC)와의 drift를 검증할 수 있어서다. 인프라 의존이 없는 순수 코드 spec이면 이 절은 통째로 생략 — FR이 필수인 것과 달리 SC·NFR·Infra는 *참조(선택)*, 팀이 원하면 --strict로 강제 승격.

제일 중요 ⭐

핵심 구조

① 1 레포 = 1 모듈

레포 하나는 정확히 모듈 하나(안정적 능력 영역, bounded context)를 담는다. "추천 서비스"가 한 모듈이면 그게 레포 하나다. 모듈 안엔 여러 spec이 산다 — 추천 로직(SPEC-001), 직원 매칭(SPEC-002), 알림(SPEC-003)… 각각 한 spec.

왜 한 레포에 여러 모듈을 안 넣나? 모듈은 독립적으로 배포·교체되는 단위여야 하기 때문이다. 한 레포에 여러 모듈을 섞으면 서로 몰래 얽혀 결국 "같이만 배포되는" 덩어리가 된다. 그래서 모듈이 늘면 레포를 나눈다.

② 한 레포 안의 두 서랍

스펙(무엇/왜)과 코드(어떻게)는 다른 디렉토리에 산다. 섞으면 안 된다.

SDD 서랍 — 무엇을 / 왜
  • sdd/specs/SPEC-001.md — FR(요구) + ## Ownership
  • sdd/MODULE_MAP.md — 이 모듈의 정체성 + spec 목록
TDD 서랍 — 어떻게
  • src/recommend.ts — production 코드
  • tests/recommend.test.ts — @covers로 FR에 연결

왜 코드를 sdd/에 안 넣나? (1) 언어 테스트러너(vitest·pytest·go test)가 평범한 소스 트리에서 테스트를 찾기 때문이고, (2) 문서(명세)와 실행(코드)이 뒤섞이면 "무엇이 진실원인가"의 경계가 무너지기 때문이다. sdd/는 문서만.

③ 둘을 잇는 사슬 — @covers FR-ID (방법론의 심장)

여기가 핵심이다. 두 서랍은 딱 하나의 끈으로 묶인다 — 요구마다 붙는 번호 FR-001, 그리고 테스트에 다는 @covers 주석. 명세에 "FR-001: 추천 반환"을 쓰면, 그 요구를 검증하는 테스트에 // @covers SPEC-001/FR-001을 단다. 그러면 "이 테스트가 저 요구를 검증한다"가 기계로 추적 가능해진다.

④ sdd.config.json — "이 프로젝트의 저장·연결 방식" 한 장

그럼 게이트는 "스펙이 어디 있고, 테스트를 어디서 찾는지"를 어떻게 아는가? 이 설정 파일 하나로 안다. 그래서 언어가 Python이든 Go든 게이트 코드를 새로 짤 필요가 없다 — 값만 바꾼다.

// 언어·접두어가 달라도 게이트 코드를 새로 짤 필요 없음 — 값만 바꾼다
{
  "specDir": "sdd/specs",                  // 스펙이 어디 사나
  "specIdPrefixes": ["SPEC","INFRA","TEST","CICD"],  // 표준 4종 접두어
  "requirementIdPrefixes": ["FR"],          // 요구 ID 접두어 — 전 파싱 사이트가 이 문법 하나를 파생
  "ownershipCategories": ["Entities","Surfaces","Capabilities"],
  "scanDirs": ["src","tests"],            // @covers를 어디서 찾나
  "commands": { "test": "vitest run" }   // 게이트가 부를 언어별 명령
}

이 한 파일이 4번째 기둥(범용성)의 실체다 — 새 언어·새 스택이 와도 게이트를 새로 만들지 않고 이 값만 맞춘다.

규모가 커지면

큰 프로그램 = 여러 모듈의 MSA 합성

추천 서비스가 커져서 결제·알림까지 필요해지면? 한 레포에 욱여넣지 않고 레포를 나눈다. 추천·결제·알림이 각각 독립 모듈-레포가 되고, 사이는 공개 계약(API/이벤트 스키마)으로만 연결된다.

강제

게이트 — "주장"을 "기계적 사실"로

여기까지가 구조다. 그런데 구조를 문서로 "이렇게 합시다"라고만 적으면 사람도 LLM도 빠뜨린다. LLM은 누락을 낸다 — 그래서 확률적 리뷰가 아니라 결정적 게이트가 1차다. 게이트는 CI(또는 git 훅)에서 돌고, 규칙을 어기면 빌드를 막는다(exit 1).

게이트무엇을 잡나강제비유

dedup ↔ cohesion = 거울상 (핵심 통찰)

방법론의 최대 빈칸이 여기 있었다. "이 spec이 기존과 중복인가?"를 판정할 규칙이 없어 사람·LLM 판단에 맡겨 뒀고 — LLM은 누락을 낸다(→ 거짓 green). 그래서 각 spec이 자기가 소유하는 키를 선언하게 하고, "하나의 키는 정확히 한 spec만 소유"라는 집합 규칙으로 바꿨다. 판정이 "감"이 아니라 "집합 조회"가 된다. (출처: 논문이 아니라 소프트웨어공학 1차 원칙 — DDD bounded context·SSOT·집합 유일성. 1차 원칙이라 결정적이고, 그래서 CI에서 exit 1로 강제할 수 있다.)

그래서 Ownership만 필수다 (SC·NFR은 선택). 이 판정은 모든 spec이 키를 선언해야 성립한다 — 안 적은 spec은 dedup 레이더 밖이라 그 중복이 안 걸린다(미선언 1개 = 보장에 뚫린 구멍). SC·NFR 누락은 그 기능 하나의 로컬 약점이지만, Ownership 누락은 시스템 전체 중복 보장을 깬다. 비유하면 Ownership은 그 spec이 점유한 땅의 등기 — 등기가 없으면 두 spec이 같은 땅에 건물을 올려도(중복) 아무도 모른다. (게이트는 미선언을 warn으로 두고 --strict로 완전 강제 — 점진 도입.)

## Ownership  ← SPEC-001이 "내가 소유한다"고 선언하는 키들
- **Entities**: pjt_projects, pjt_staff       # 도메인 객체/테이블
- **Surfaces**: POST /api/recommend           # route·화면·이벤트
- **Capabilities**: staff.recommend           # Entity×Action

같은 ## Ownership 데이터를 양방향으로 보면 정반대의 두 문제를 같은 도구로 잡는다:

2계층(정직). 게이트는 "키가 같은" 구조적 중복만 막는다. "말만 다른 같은 요구"(의미적 중복)는 못 잡는다 → 같은 Entity를 소유한 이웃 spec과만 좁혀서 LLM 리뷰. 100% 자동화는 불가하니, 게이트(구조)+좁힌 리뷰(의미) 2계층으로 닫는다. 단, 절차와 어휘는 문법화했다: 검토했다는 기록(## Dedup-Review — 검토한 이웃 spec ID+판정)은 completeness가 존재와 참조 실재(언급한 ID가 실재하는 spec인가)를 검사하고, entity 어휘는 entityRegistry 등록제(신설 = config 리뷰 관문)가 막는다 — 판정의 질만 사람 몫으로 남는다. 이렇게 "게이트가 판정하지 않는 것"은 전부 선언된 리뷰 경계다(METHODOLOGY.md 리뷰 경계 선언 표 — 정의되지 않은 예외 0).

"SC·인수조건이 빠진 것 같다" — 왜 그렇게 보이고, 실제로는

이 키트로 직접 만들다 보면 SC와 인수조건이 사라진 것처럼 느껴진다. 맞다 — 위 표에서 보듯 게이트·@covers 사슬·하네스가 전부 FR 중심이라 그렇게 보인다. 하지만 빠진 게 아니라 담당이 나뉘어 있다:

이 키트가 강제 — FR
  • FR ↔ test ↔ code 추적 (@covers · fr-coverage)
  • 중복 · 입도 · 완전성 (ownership · cohesion · completeness)
Spec Kit 네이티브가 담당 — 인수조건 · SC
  • 인수조건(GWT) → /tasks로 task·test화, /analyze 커버리지
  • SC 측정가능성 → /checklist

그래서 키트는 check-spec-completeness로 "FR 있는 spec엔 SC·인수조건이 존재하는가"를 advisory로 지켜 둘을 그늘에서 끌어낸다. 단 SC의 충족(예: p95 < 200ms를 실제 달성했나)은 런타임 지표라 빌드 게이트가 강제하지 않는다 — 못 하는 걸 한다고 하지 않고 정직하게 둔다. 요약: 빠진 게 아니라, FR은 키트가·인수조건/SC는 Spec Kit이 맡고, 존재는 게이트가 지킨다.

기능 하나가 흐르는 길

생애주기 — 0~8 루프

"직원 추천"을 새로 만든다면, 이 순서로 흐른다. 한 칸 한 칸이 게이트를 통과해야 다음으로 간다.

1specify"직원 추천" 한 마디 → spec (FR을 EARS로)
2clarify빈칸([NEEDS CLARIFICATION]) 질문해서 메움
3planFR ↔ 컴포넌트 매핑
4tasks2~5분 TDD 작업으로 분해
5analyze누락·모순·중복 + 게이트(ownership/cohesion)
6구현TDD RED→GREEN — 테스트 먼저, 통과시킴
7머지사람 PR 리뷰·승인
8converge시간 지나 drift났나 검사 → 갭 표면화
9close-out트래커 유래 작업이면 — 트래커 dev-done → 이해관계자 보고 → 리포터 confirm

양방향 동기화 — 작성은 LLM, 승인은 사람.
· 위→아래(spec→code): 사람이 spec 작성 → 미구현을 구현.
· 아래→위(code→spec, 핫픽스 후): 코드가 먼저 바뀜 → 사람이 "무슨 기능·왜"(intent) 한 줄 입력 → LLM이 코드+intent로 spec 갱신 → 사람이 "정본화 vs 되돌리기" 승인. — 코드만 보고 spec을 자동으로 덮어쓰지 않는다(그러면 spec이 코드의 거울로 전락해 drift 탐지가 무의미해진다).

⑨ 완료 루프의 꼬리 — 원점 트래커 close-out. 실무 작업의 상당수는 QA/이슈 트래커에서 유래한다(예: 페이지별 QA 메모, open→done→confirmed). 그런 작업은 verify/merge로 끝이 아니라 — ①트래커 dev-done(개발자) → ②이해관계자 완료 보고(무엇·왜·어떻게+검증 경로) → ③리포터 confirm(리포터/QA)까지가 완료다. 2인 책임분리: 개발자는 confirm을 건드리지 않는다. 트래커·보고 채널은 킷에 하드코딩하지 않고 trackerCloseout config로 인스턴스화 — 외부 시스템·사람 몫이라 게이트가 아니라 규범이다.

하네스 — 이 모든 걸 "수시로" 돌린다

지금까지는 규칙·구조였다. 하지만 규칙이 있다고 사람이 매번 "아 이제 게이트 돌리고, 중복 보고, 명세 맞춰야지"를 기억할까? 안 한다. 하네스는 그 운영을 사람과 대화하며 자동으로 끌고 가는 층이다. 핵심은 — 새 판정 로직을 만드는 게 아니라, 이미 있는 게이트를 트리거하고(detect), 사람에게 묻고(ask), 행동을 잇는다(act).

결정성 — 키·경계·PREFIX

Ownership 키의 결정성 — 누가 만들어도 같은 문자열

게이트는 "한 키 = 한 spec"을 비교한다. 그런데 그 키가 어떻게 만들어지는가는 규칙 없이 사람·LLM 판단에 맡겨져 있었다 — "같은 개념인데 표기가 달라" 거짓 green이 나오거나, "다른 개념인데 비슷해 보여" 차단이 누락됐다. 이 결정성 설계가 메우는 빈칸이다.

키 생성 결정 절차 — 사람·LLM이 밟으면 같은 문자열

1 spec = 1 aggregate (분리의 결정적 기준)

"무엇이 하나의 spec인가"도 결정적 기준이 필요하다. "응집 묶음"은 주관적 — 대신 aggregate root(독립적으로 생성·삭제되는 핵심 Entity)를 경계 닻으로 쓴다.

소유

한 spec = 한 aggregate root를 Ownership.Entities에 소유. 그 aggregate를 변경하는 Capability·Surface도 함께.

참조

다른 aggregate는 Dependencies로 참조(dedup 제외). 예: recommendation spec이 staff를 읽을 때.

라우팅

새 FR → 그 FR이 변경하는 aggregate 식별 → 그 owner spec 개정. owner 없으면 새 spec(새 aggregate).

세 게이트가 한 닻으로 묶인다: dedup(1 Entity=1 spec = 경계 강제) · cohesion(Entities 2개 이상 = 여러 aggregate 삼킴 경고) · Dependencies(경계 밖 참조 명시). "독립 생성·삭제되는 root"라는 조작적 정의가 "응집 묶음"보다 훨씬 결정적이고, dedup이 사후에 강제한다.

실전 예시 — 세 키의 역할 · 라우팅 · 중복 방지

추천 모듈의 SPEC-001 ## Ownership이 이렇다고 하자:

## Ownership
- Entities:     recommendation                               ← aggregate root: "무엇에 대한 spec인가" (소유 경계 · 새 spec의 유일한 근거)
- Surfaces:     POST /api/recommend/{id}, GET /api/recommend  ← 진입점: "어디로 들어오나" (route · 화면 · job · event)
- Capabilities: recommendation.create, recommendation.list   ← 동작: "그 Entity에 무슨 일" (entity × verb)
## Dependencies
- Entities:     staff                                        ← 읽기/호출만 (소유 아님 → dedup 제외)

세 키의 역할: Entity = 정체·경계(aggregate root — 새 spec을 정당화하는 유일한 키) · Surface = 진입점(그 기능에 닿는 route/job/event) · Capability = 동작(Entity×verb). Surface·Capability는 Entity에 매달리므로 그 자체로는 새 spec 사유가 아니다 — 오직 새 Entity(aggregate)만 새 spec을 낳는다.

새 요구키 판정착지

그래서 스펙 중복이 어떻게 막히나 — 3단:

① 결정성(정규화)

표기 위장 중복 차단

누가 Recommendation.Create·:id로 적어도 정규화가 recommendation.create·{id}로 접는다 → "같은 개념 = 같은 문자열". 표기만 다른 중복이 조용히 두 spec으로 갈라지지 못한다.

② 소유 조회 라우팅

③ dedup 게이트

실수까지 기계 차단

그래도 SPEC-003이 recommendation.create를 또 선언하면 check-ownership이 exit 1(한 키 = 한 spec). 사람이 놓쳐도 CI가 막는다.

핵심: 중복 방지는 사람의 기억이 아니라 결정적 키(정규화) → 소유 레지스트리 조회 → dedup 게이트의 3단이다. 그래서 spec이 수십 개로 늘어도 "같은 기능을 두 번 명세"가 집합 멤버십으로 기계 차단된다. Entity가 유일한 새-spec 근거라는 점이 이 전체를 결정적으로 만든다 — Surface·Capability로는 spec이 갈라지지 않으니까.

Entity 관계 — 쪼갠 aggregate 사이의 참조를 구조화 (SPEC-017)

"1 spec = 1 aggregate"는 원칙이지만 실사용은 자주 무너진다(도그푸딩 실측: 소비 프로젝트의 한 spec이 aggregate root 7개를 안고 있어 cohesion 상한을 넘김). 과대 spec을 aggregate별로 쪼개면, 쪼개진 Entity들 사이의 관계를 적을 문법이 필요하다 — 지금까지 Dependencies.Entities는 완전한 자유 텍스트라 대상의 실재·소유 spec·순환 여부를 게이트가 전혀 보지 못했다.

구조화(opt-in)

EntityName (relation-type) 괄호 표기만 구조화 관계로 인정 — 소문자 kebab 1토큰만. 괄호 없이 이름만 적으면 지금처럼 검사 없는 레거시 자유참조(하위호환).

실재·소유 spec = hard

대상 Entity를 전체 spec의 Ownership에서 자동 해석 — 못 찾으면(오타·삭제) exit 1. 소유 spec을 수동으로 안 적어도 되니 재번호·리네임에 강하다.

순환 참조 = advisory

A→B→A처럼 aggregate 간 참조가 한 방향이 아니면 경고만(차단 안 함) — 실사용에서 항상 잘못된 건 아니라서.

spec-first 강제

코드 변경은 반드시 소유 스펙을 동반한다

핫픽스 하나가 "급해서" 스펙 없이 커밋되면, 도면과 건물이 어긋난다. spec-first 게이트는 코드를 바꿀 때 같은 changeset(브랜치) 안에서 소유 스펙이 함께 갱신됐는지 commit-msg 훅에서 hard 차단한다. "소유"는 스펙의 Files glob으로 선언된다.

Files 소유매핑 — 코드 경로 → glob → 스펙

각 스펙의 ## Ownership 절에 - **Files**: <glob> 한 줄을 추가한다. 이것이 "이 코드는 이 스펙이 소유한다"는 선언이다.

커밋 분류 — 3가지로 나뉜다

커밋 종류Files glob 매치스펙 동반 필요결과
기능 변경 — spec 소유 코드 수정✓ 매치필수FR/Edge Cases/Change Log 중 하나 추가
hotfix — 버그 픽스(스펙 진짜 무관)✓ 매치트레일러 허용커밋 메시지에 Spec-Impact: none <사유>
무관 파일 — 설정·빌드 등 Files 미매치✗ 미매치불필요침묵 통과

changeset = 브랜치(top-down 흐름 보존). staged(현 커밋) ∪ base...HEAD(이전 커밋들)를 합집합으로 본다 — 스펙을 먼저 커밋하고 코드를 나중에 커밋해도 같은 브랜치면 통과. "코드를 커밋할 때마다 스펙도 같이 고쳐야"가 아니라 "브랜치 안에서 어느 커밋에든 스펙이 있으면 된다".

commit-msg 훅 실측 출력

소유 코드만 스테이징하고 스펙은 건드리지 않은 채 커밋하면:

$ git commit -m "fix: improve pdf parsing" — FAIL 실측

· spec-sync: base(origin/main) 해석 불가 — staged만 판정(경고)
spec-sync 게이트 — mode:staged(hard) changed:1 specs:1
  ✗ src/lib/pdf/parse.ts → 소유 스펙 SPEC-001에 의미 있는 변경 없음(FR/Edge Cases/Change Log)

✗ spec-first 위반: 소유 스펙을 같은 changeset에 갱신하라 — /speckit.fix 사용.
  · 스펙을 이미 수정했다면 `git add`로 스테이징했는지 확인(§6.2).
  · 진짜 스펙 무관이면 커밋 메시지에 `Spec-Impact: none <사유>` 트레일러.

소유 스펙의 Change Log에 새 행을 추가한 후 재실행하면:

$ git add sdd/specs/SPEC-001.md && git commit -m "fix: ..." — PASS 실측

· spec-sync: base(origin/main) 해석 불가 — staged만 판정(경고)
spec-sync 게이트 — mode:staged(hard) changed:2 specs:1
spec-sync: OK — 소유 코드 변경에 스펙 동반됨(또는 대상 없음).

/speckit.fix — 버그픽스 SDD 경로

spec-first 차단이 뜨면 /speckit.fix 스킬이 해소를 안내한다:

1소유 스펙 확인변경 파일이 어느 spec의 Files glob에 속하는지
2버그 성격 판단스펙 FR을 고치는 변경인가, 구현 버그인가
3스펙 갱신FR 수정 또는 Edge Cases/Change Log 행 추가
4git add + 재커밋스펙 파일도 스테이징해야 통과

스펙과 진짜 무관한 변경(리팩터링·포맷팅·설정 파일)은 커밋 메시지에 Spec-Impact: none <사유> 트레일러를 달면 즉시 통과. 사유 없이 Spec-Impact: none만 쓰면 차단(사유 필수).

스펙 수명주기 — 리뷰 안 된 스펙은 코드를 이끌 수 없다

스펙 헤더 Status:는 enum이다 — Draft → Reviewed → Approved → Active → Deprecated → Removed. 강제하는 것은 시간 순서가 아니라 상태 순서다:

1Draft작성 중 — 이 스펙이 소유한 코드 변경은 commit-msg가 차단(스펙을 같이 고쳐도)
2Reviewed/analyze+/checklist 결과를 ## Review Log(일시·수행자·판정)와 ## Dedup-Review(이웃 검토)에 기록해야 승격 — 기록 존재를 completeness가 검사
3Approved → Active승인 후 구현 — 이때부터 소유 코드가 자유롭게 흐른다(동반 변경 규칙은 그대로)

미소유 파일 정책(specSyncUnownedPolicy): 어느 스펙의 Files에도 안 잡히는 변경 파일은 원래 침묵 통과였다 — 이제 silent(현행)/warn/error(closed-world) 중 하나를 선언한다. 의도적 예외는 specSyncExemptGlobs로. 검증 회계(requireAccounting): 커버 안 된 FR이 경고 속에 쌓이는 것도 막는다 — 모든 FR은 unit(@covers) ∨ smoke(매니페스트 evidence) ∨ deferred(매니페스트 reason)로 회계되어야 하고, 게이트는 사유의 존재만 강제한다(질은 리뷰 몫). Status 없는 레거시 스펙·기본값 config는 전부 현행 동작 그대로다(점진 도입 — advisory→strict 승격 경로).

Draft 차단의 사각지대 — 웹 UI 병합은 로컬 훅을 안 탄다 (draftBlockPolicy, SPEC-008 FR-007)

위 Draft 차단은 설계상 --staged(commit-msg 훅) 모드에서만 hard고, range 모드는 항상 advisory였다. 그런데 GitHub/GitLab의 웹 UI "Merge" 버튼은 로컬 git 훅을 절대 실행하지 않는다 — 로컬 클론에 훅이 설치돼 있어도 무관하다. 도그푸딩에서 실측: Draft 상태 spec이 소유한 Jenkinsfile이 Merge branch '...' into 'main' 병합 커밋으로 여러 차례 새어나갔고, 그 리포의 CI엔 게이트 호출 자체가 없어 range advisory조차 돌지 않았다. "채택 = 상시 강제"가 실제로는 로컬 커밋 경로에만 성립했던 것.

draftBlockPolicy: "advisory"(기본)

하위호환 — range 모드는 지금처럼 경고만, exit 0.

draftBlockPolicy: "hard"

range 모드에서도 Draft 위반이 exit 1. CI가 <GATE> specsync [base]를 MR 파이프라인에 걸면, 로컬 훅이 안 타는 웹 UI 병합도 이 지점에서 막힌다.

로컬 pre-push/commit-msg 훅과 이 CI 스텝은 서로를 대체하지 않는다 — 훅은 로컬 커밋 경로의 백스톱, 이 스텝은 훅이 절대 닿지 않는 서버측 병합 경로의 백스톱이다. 둘 다 있어야 "채택 = 상시 강제"가 실제로 완성된다.

재도출 소스 회계 — 재생성은 src만 읽으면 미완성이다

두 프로젝트 재생성 비교에서 [검증]된 결론: 재생성은 레포 안에서 도출 가능한 모든 것을 손작성본보다 잘 뽑지만, 초기 재도출이 src 밖(terraform/k8s/CI)을 아예 안 읽어 INFRA FR을 잃었고, smoke 실증거·@covers 재연결·인간 의도는 수작업으로 남았다. 그 4개 잔여를 전부 문법화했다:

1소스 9클래스 회계sdd/derivation.json에 code·iac·ci·ops-docs·build-evidence·vcs-history·prior-traceability·prior-intent·human-intent 전부를 mapped ∨ none ∨ deferred로 선언 — derivation 게이트가 미회계·"실재하는데 none"을 차단. 착지도 강제: iac/ci에서 도출한 스펙이 클래스 접두어(iac→INFRA-·ci→CICD-)가 아니면 fr 게이트의 접두어↔클래스 정합이 차단(§PREFIX 거버넌스)
2smoke 증거 자동 수집증거가 사는 파일(CI 정의·스크립트·runbook)에 @verifies 태그 → smoke-scan --write가 매니페스트를 결정적 재생성(수동 연결 제거, 드리프트는 check가 차단)
3태그 재연결 자동화재도출은 FR 키 보존이 기본 — 재번호가 불가피하면 마이그레이션 맵 + retag --write가 @covers/@verifies/매니페스트 키를 기계 이행(손 재태깅 금지)
4선제 캡처어디에도 기록 안 된 순수 의도는 사후 재도출 불가 — Review Log·Dedup-Review·Clarifications에 더해 Change Log 근거 칸의 존재를 completeness가 검사(빈 근거 = 경고, --strict 차단)

정직한 경계 = 선언된 클래스. 레포 밖 실체(운영 인프라 체인·빌드 로그)는 게이트가 아는 척하지 않는다 — mapped evidence가 좌표(빌드 #·URL·describe 명령)를 가리키는 방식으로만 선언되고, 그 좌표 검증은 SSOT.md §5b(live drift)와 리뷰 몫이다. 순수 인간 의도(human-intent)도 "정의되지 않은 예외"가 아니라 회계되는 클래스다 — 저술 시점 캡처만이 경로임을 매니페스트가 명시한다.
