Spec-Driven Development · 범용 키트

"무엇·왜"와 "어떻게"를
항상 일치시키는 개발 방법

스펙(무엇을·왜)과 코드(어떻게)를 다른 서랍에 나눠 두고, 둘이 어긋나지 않도록 기계(게이트)가 강제하고 사람이 승인한다. Spec Kit · EARS · Superpowers를 한 골격으로 묶고, 언어·모델·인프라에 무관하게 같게 돈다.

비유 — 건축 도면과 건물. 도면(spec) 없이 건물(code)을 올리면 "왜 이렇게 지었지?"를 아무도 모른다. 도면만 그리고 건물과 안 맞으면 도면이 거짓말이 된다. 이 방법론은 도면과 건물을 묶어두고, 어긋나면 경보가 울리게 한다.

이 글은 하나의 가상 기능 — "프로젝트에 직원을 추천해주는 기능" — 을 처음부터 끝까지 따라가며 설명한다. 추천 서비스가 한 모듈, 그 안의 SPEC-001이 그 기능의 명세다.

왜 필요한가

AI는 코드를 잘 짠다 — 무너지는 건 그 주변이다

LLM은 구현엔 강하다. "직원 추천 함수 짜줘"라고 하면 잘 짠다. 문제는 그 코드를 둘러싼 것들이다. 한 달 뒤 "이 추천 로직 왜 이렇게 됐지?", "이거 이미 다른 데서 만들지 않았나?", "명세엔 A인데 코드는 B인데?" — 이 셋이 시간이 지나며 반드시 무너진다. 이 방법론은 그걸 구조 + 기계적 강제로 막는 것이 전부다.

① 흩어짐

요구사항이 채팅·머릿속·주석에 흩어져 단일 진실원(SSOT)이 없다. "무엇을 만들기로 했더라?"에 답할 곳이 없다.

② 표류(drift)

핫픽스로 코드만 고치고 명세는 그대로 둔다 → 시간이 지나면 도면과 건물이 따로 논다. 명세가 거짓말이 된다.

③ 중복

이미 "추천" 기능이 있는 줄 모르고 비슷한 걸 또 만든다. 진실이 두 곳으로 갈라진다.

무엇을 조합했나

4개의 기둥

혼자 만든 게 아니라, 검증된 도구 셋을 한 골격으로 묶고 거기에 "어디서나 같게 돈다"는 보장을 더했다. 앞 3개가 무엇을 하는가, 4번째가 어디서나 같게 돈다는 약속이다.

골격

GitHub Spec Kit

specify→plan→tasks 흐름 + 리뷰 명령. "스펙의 옳음"을 담당. 30개+ 에이전트 비종속.

요구 형식

EARS

요구를 모호하지 않은 5가지 문장틀로 강제. "한 요구 = 한 동작 = 검증가능".

구현·검증

Superpowers

TDD(테스트 먼저)·완료전검증·코드리뷰. "코드의 옳음"을 담당.

보장

범용성

언어·모델·DB·클라우드·CI 무엇이든 같게. 차이는 설정 한 장으로만 흡수.

왜 한 골격으로 묶나 — 셋은 서로의 구멍을 메운다

따로 쓰면 각자 한 부분만 잘하고 구멍이 남는다. 묶어야 무엇을(EARS로 정밀히) → 관리(Spec Kit) → 검증(Superpowers) 사슬이 닫힌다 — 하나라도 빠지면 끊긴다.

혼자면

Spec Kit 단독

spec을 관리하지만 FR이 모호해도 못 막고, 코드가 맞는지 검증하지 않는다.

혼자면

EARS 단독

요구를 검증 가능하게 만들지만 문장 형식일 뿐 — 관리도 검증도 못 한다.

혼자면

Superpowers 단독

코드를 검증하지만 "무엇이 명세인가·중복인가"를 모른다.

EARS 없으면 FR이 모호해 테스트로 못 묶고, Spec Kit 없으면 명세가 흩어지고, Superpowers 없으면 코드가 미검증이라 명세가 희망사항이 된다 — 그래서 한 골격. (+ 4번째 기둥 범용성이 어디서나 같게 돌린다.)

EARS 5패턴 — 우리 예제로 보면

"직원 추천"의 요구를 EARS로 쓰면 모호함이 사라진다. 예: WHEN 직원 조회 요청이 오면, THE SYSTEM SHALL 적합도 순으로 추천 목록을 반환한다.

패턴문장틀
UbiquitousTHE SYSTEM SHALL <응답> (항상)
EventWHEN <트리거>, THE SYSTEM SHALL <응답>
StateWHILE <상태>, THE SYSTEM SHALL …
UnwantedIF <비정상>, THEN THE SYSTEM SHALL …
OptionalWHERE <옵션>, THE SYSTEM SHALL …

금지: "적절히", "가능하면", should, 한 문장에 동작 2개, 측정 불가. — 모호하면 검증할 수 없고, 검증 못 하면 게이트가 못 잡는다.

⚠ EARS는 spec 전체가 아니라 FR 절에만 적용된다. 한 spec은 여러 절로 구성되고 절마다 형식이 다르다 — 그리고 코드와 @covers로 묶이는 추적 단위는 FR뿐이다. SC·NFR은 검증·측정의 기준이지 코드에 1:1로 매달리지 않는다.

spec의 절형식출처코드 연결
User Story / 시나리오서술 문단Spec Kit 기본—
AcceptanceGiven–When–ThenSpec Kit 기본—
FR (Functional Req.)EARS 5패턴절=기본 · EARS=우리 preset@covers → test↔code
SC (Success Criteria)측정형 숫자 (예: 추천 정확도 ≥ 80%)Spec Kit 기본—
NFR (Non-Functional)품질 제약 (성능·보안·가용성, 예: p95 < 200ms)우리 템플릿—
Ownership소유 키 (Entities/Surfaces/Capabilities)우리 추가게이트 dedup·cohesion

분담: FR은 이 키트의 게이트(@covers·dedup·cohesion·completeness)가 추적·강제하고, 인수조건·SC는 Spec Kit 네이티브(/analyze·/checklist·/tasks)가 담당한다. check-spec-completeness는 SC·인수조건이 존재하는지만 advisory로 지킨다 — SC 충족은 런타임 지표라 빌드 게이트가 강제할 수 없다.

선택 절 Infrastructure Prerequisites의 "왜": 인프라 의존도 요구사항이라 spec에 적되 역량으로 적는다 — "내구성 있는 이벤트 로그 필요"(O) / "Kafka"(X). spec에 두는 이유는 그 의존이 추적되고, 배포 실제(IaC)와의 drift를 검증할 수 있어서다. 인프라 의존이 없는 순수 코드 spec이면 이 절은 통째로 생략 — FR이 필수인 것과 달리 SC·NFR·Infra는 *참조(선택)*, 팀이 원하면 --strict로 강제 승격.

제일 중요 ⭐

핵심 구조

① 1 레포 = 1 모듈

레포 하나는 정확히 모듈 하나(안정적 능력 영역, bounded context)를 담는다. "추천 서비스"가 한 모듈이면 그게 레포 하나다. 모듈 안엔 여러 spec이 산다 — 추천 로직(SPEC-001), 직원 매칭(SPEC-002), 알림(SPEC-003)… 각각 한 spec.

왜 한 레포에 여러 모듈을 안 넣나? 모듈은 독립적으로 배포·교체되는 단위여야 하기 때문이다. 한 레포에 여러 모듈을 섞으면 서로 몰래 얽혀 결국 "같이만 배포되는" 덩어리가 된다. 그래서 모듈이 늘면 레포를 나눈다.

② 한 레포 안의 두 서랍

스펙(무엇/왜)과 코드(어떻게)는 다른 디렉토리에 산다. 섞으면 안 된다.

SDD 서랍 — 무엇을 / 왜
  • sdd/specs/SPEC-001.md — FR(요구) + ## Ownership
  • sdd/MODULE_MAP.md — 이 모듈의 정체성 + spec 목록
TDD 서랍 — 어떻게
  • src/recommend.ts — production 코드
  • tests/recommend.test.ts — @covers로 FR에 연결

왜 코드를 sdd/에 안 넣나? (1) 언어 테스트러너(vitest·pytest·go test)가 평범한 소스 트리에서 테스트를 찾기 때문이고, (2) 문서(명세)와 실행(코드)이 뒤섞이면 "무엇이 진실원인가"의 경계가 무너지기 때문이다. sdd/는 문서만.

③ 둘을 잇는 사슬 — @covers FR-ID (방법론의 심장)

여기가 핵심이다. 두 서랍은 딱 하나의 끈으로 묶인다 — 요구마다 붙는 번호 FR-001, 그리고 테스트에 다는 @covers 주석. 명세에 "FR-001: 추천 반환"을 쓰면, 그 요구를 검증하는 테스트에 // @covers SPEC-001/FR-001을 단다. 그러면 "이 테스트가 저 요구를 검증한다"가 기계로 추적 가능해진다.

④ sdd.config.json — "이 프로젝트의 저장·연결 방식" 한 장

그럼 게이트는 "스펙이 어디 있고, 테스트를 어디서 찾는지"를 어떻게 아는가? 이 설정 파일 하나로 안다. 그래서 언어가 Python이든 Go든 게이트 코드를 새로 짤 필요가 없다 — 값만 바꾼다.

// 언어·접두어가 달라도 게이트 코드를 새로 짤 필요 없음 — 값만 바꾼다
{
  "specDir": "sdd/specs",                  // 스펙이 어디 사나
  "specIdPrefixes": ["SPEC"],               // 어떤 접두어가 1급 스펙인가
  "ownershipCategories": ["Entities","Surfaces","Capabilities"],
  "scanDirs": ["src","tests"],            // @covers를 어디서 찾나
  "commands": { "test": "vitest run" }   // 게이트가 부를 언어별 명령
}

이 한 파일이 4번째 기둥(범용성)의 실체다 — 새 언어·새 스택이 와도 게이트를 새로 만들지 않고 이 값만 맞춘다.

규모가 커지면

큰 프로그램 = 여러 모듈의 MSA 합성

추천 서비스가 커져서 결제·알림까지 필요해지면? 한 레포에 욱여넣지 않고 레포를 나눈다. 추천·결제·알림이 각각 독립 모듈-레포가 되고, 사이는 공개 계약(API/이벤트 스키마)으로만 연결된다.

강제

게이트 — "주장"을 "기계적 사실"로

여기까지가 구조다. 그런데 구조를 문서로 "이렇게 합시다"라고만 적으면 사람도 LLM도 빠뜨린다. LLM은 누락을 낸다 — 그래서 확률적 리뷰가 아니라 결정적 게이트가 1차다. 게이트는 CI(또는 git 훅)에서 돌고, 규칙을 어기면 빌드를 막는다(exit 1).

게이트무엇을 잡나강제비유
check-fr-coverage테스트 없는 FR (사슬 끊김)차단도면엔 있는데 안 지은 방
check-ownership (dedup)2개 spec이 같은 키 소유 = 중복차단같은 방을 두 도면이 그림
check-spec-cohesion (거울상)1개 spec이 키/FR 과다 = 욱여넣기권고한 도면에 집 전체를 우겨넣음
check-spec-completenessFR 있는데 SC·인수조건 없음권고도면에 합격 기준이 안 적힘
보강 3종빈 테스트 / 코드만 바뀜 / 명세 없는 코드권고advisory 경고

dedup ↔ cohesion = 거울상 (핵심 통찰)

방법론의 최대 빈칸이 여기 있었다. "이 spec이 기존과 중복인가?"를 판정할 규칙이 없어 사람·LLM 판단에 맡겨 뒀고 — LLM은 누락을 낸다(→ 거짓 green). 그래서 각 spec이 자기가 소유하는 키를 선언하게 하고, "하나의 키는 정확히 한 spec만 소유"라는 집합 규칙으로 바꿨다. 판정이 "감"이 아니라 "집합 조회"가 된다. (출처: 논문이 아니라 소프트웨어공학 1차 원칙 — DDD bounded context·SSOT·집합 유일성. 1차 원칙이라 결정적이고, 그래서 CI에서 exit 1로 강제할 수 있다.)

그래서 Ownership만 필수다 (SC·NFR은 선택). 이 판정은 모든 spec이 키를 선언해야 성립한다 — 안 적은 spec은 dedup 레이더 밖이라 그 중복이 안 걸린다(미선언 1개 = 보장에 뚫린 구멍). SC·NFR 누락은 그 기능 하나의 로컬 약점이지만, Ownership 누락은 시스템 전체 중복 보장을 깬다. 비유하면 Ownership은 그 spec이 점유한 땅의 등기 — 등기가 없으면 두 spec이 같은 땅에 건물을 올려도(중복) 아무도 모른다. (게이트는 미선언을 warn으로 두고 --strict로 완전 강제 — 점진 도입.)

## Ownership  ← SPEC-001이 "내가 소유한다"고 선언하는 키들
- **Entities**: pjt_projects, pjt_staff       # 도메인 객체/테이블
- **Surfaces**: POST /api/recommend           # route·화면·이벤트
- **Capabilities**: staff.recommend           # Entity×Action

같은 ## Ownership 데이터를 양방향으로 보면 정반대의 두 문제를 같은 도구로 잡는다:

2계층(정직). 게이트는 "키가 같은" 구조적 중복만 막는다. "말만 다른 같은 요구"(의미적 중복)는 못 잡는다 → 같은 Entity를 소유한 이웃 spec과만 좁혀서 LLM 리뷰. 100% 자동화는 불가하니, 게이트(구조)+좁힌 리뷰(의미) 2계층으로 닫는다.

"SC·인수조건이 빠진 것 같다" — 왜 그렇게 보이고, 실제로는

이 키트로 직접 만들다 보면 SC와 인수조건이 사라진 것처럼 느껴진다. 맞다 — 위 표에서 보듯 게이트·@covers 사슬·하네스가 전부 FR 중심이라 그렇게 보인다. 하지만 빠진 게 아니라 담당이 나뉘어 있다:

이 키트가 강제 — FR
  • FR ↔ test ↔ code 추적 (@covers · fr-coverage)
  • 중복 · 입도 · 완전성 (ownership · cohesion · completeness)
Spec Kit 네이티브가 담당 — 인수조건 · SC
  • 인수조건(GWT) → /tasks로 task·test화, /analyze 커버리지
  • SC 측정가능성 → /checklist

그래서 키트는 check-spec-completeness로 "FR 있는 spec엔 SC·인수조건이 존재하는가"를 advisory로 지켜 둘을 그늘에서 끌어낸다. 단 SC의 충족(예: p95 < 200ms를 실제 달성했나)은 런타임 지표라 빌드 게이트가 강제하지 않는다 — 못 하는 걸 한다고 하지 않고 정직하게 둔다. 요약: 빠진 게 아니라, FR은 키트가·인수조건/SC는 Spec Kit이 맡고, 존재는 게이트가 지킨다.

기능 하나가 흐르는 길

생애주기 — 0~8 루프

"직원 추천"을 새로 만든다면, 이 순서로 흐른다. 한 칸 한 칸이 게이트를 통과해야 다음으로 간다.

1specify"직원 추천" 한 마디 → spec (FR을 EARS로)
2clarify빈칸([NEEDS CLARIFICATION]) 질문해서 메움
3planFR ↔ 컴포넌트 매핑
4tasks2~5분 TDD 작업으로 분해
5analyze누락·모순·중복 + 게이트(ownership/cohesion)
6구현TDD RED→GREEN — 테스트 먼저, 통과시킴
7머지사람 PR 리뷰·승인
8converge시간 지나 drift났나 검사 → 갭 표면화

양방향 동기화 — 작성은 LLM, 승인은 사람.
· 위→아래(spec→code): 사람이 spec 작성 → 미구현을 구현.
· 아래→위(code→spec, 핫픽스 후): 코드가 먼저 바뀜 → 사람이 "무슨 기능·왜"(intent) 한 줄 입력 → LLM이 코드+intent로 spec 갱신 → 사람이 "정본화 vs 되돌리기" 승인. — 코드만 보고 spec을 자동으로 덮어쓰지 않는다(그러면 spec이 코드의 거울로 전락해 drift 탐지가 무의미해진다).

조종석

하네스 — 이 모든 걸 "수시로" 돌린다

지금까지는 규칙·구조였다. 하지만 규칙이 있다고 사람이 매번 "아 이제 게이트 돌리고, 중복 보고, 명세 맞춰야지"를 기억할까? 안 한다. 하네스는 그 운영을 사람과 대화하며 자동으로 끌고 가는 층이다. 핵심은 — 새 판정 로직을 만드는 게 아니라, 이미 있는 게이트를 트리거하고(detect), 사람에게 묻고(ask), 행동을 잇는다(act).

규칙언제(trigger)무엇을 묻나(ask) → 행동(act)
R1 spec→codespec 바뀜"이 FR들 코드 만들/고칠까요?" → TDD 구현
R2 code→spec코드만 바뀜"기존 고칠까 / 새로 만들까 / 무시?" → specify 갱신
R3 dedup+입도spec 생긴 직후"중복 합칠까 / 과대 spec 쪼갤까?" → 통합·분할
R4 상시 syncpush·수시위를 일괄 점검 → drift면 해당 규칙으로 라우팅
실전 — 하네스가 실제로 도는 모습

핫픽스 한 줄이 명세까지 도달하기까지

추상적인 루프 말고, 실제 시나리오로 보자. 가장 흔하고 위험한 경우 — 급해서 코드부터 고친 핫픽스(아래→위, R2 + R4)다. "주말엔 추천을 끄자"를 개발자가 코드에 바로 박았다고 하자.

개발자code
1. 코드부터 고침

src/recommend.ts에 "주말 제외" 로직을 추가하고 커밋. 명세는 안 건드림. (전형적 핫픽스 — 여기서 drift가 태어난다.)

pre-push훅
2. 밀려는 순간 자동 점검

git push → pre-push 훅이 node scripts/sdd-sync.mjs를 돌린다. (R4 "수시"의 값싼 1차 — 밀 때마다.)

하네스detect
3. drift 감지 → 안내(비차단)

check-converge-drift가 "코드는 바뀌었는데 spec은 그대로"를 잡는다. 리포트를 찍고 "/sdd-sync 돌려라"라고 안내한다 — push는 막지 않는다.

$ git push → pre-push 훅 출력

SDD sync 리포트 — detector 일괄 실행 (HARNESS.md 규칙표)

● R1 spec→code: ✓ clean
● R2 code→spec: ⚠ 확인 필요
    [check-converge-drift.mjs] · 코드 1건 변경인데 스펙 무변경
● R3 dedup+입도+완전성: ✓ clean

요약: 확인 필요 — R2 code→spec → '/sdd-sync'로 의사결정
↑ spec↔code drift 가능 — '/sdd-sync'로 정렬 검토. (push는 계속됨)
하네스ask
4. 개발자가 /sdd-sync 실행 → 하네스가 물음

"코드만 바뀐 부분이 있습니다(R2). ① 기존 SPEC-001 개정 / ② 새 spec / ③ 의도적 무시(hotfix) — 어느 쪽인가요?"

사람결정
5. 사람이 결정 + intent 입력

"① 기존 SPEC-001 개정"을 고르고 intent 한 줄: "주말엔 운영 부하를 피하려 추천을 끈다".

하네스act
6. LLM이 명세를 갱신(작성)

/speckit.converge로 갭 표면화 → /speckit.specify update가 코드 diff + intent로 FR-007: WHILE 주말, THE SYSTEM SHALL 추천을 비활성화한다를 SPEC-001에 추가 → /speckit.analyze로 정합 검사.

하네스재검증
7. 다시 점검 → 새 경보가 뜰 수도

/sdd-sync 재실행: R2는 이제 clean. 단 SPEC-001이 FR이 많아졌다면 check-spec-cohesion이 "이 spec 과대 — 쪼갤까?"(R3)를 물을 수 있다. → 사람이 다시 결정.

사람승인
8. bless(정본화)

사람이 명세 변경을 승인한다. 끝 — 도면과 건물이 다시 일치한다. 작성은 LLM이 했지만, 무엇을 진실로 삼을지는 사람이 정했다.

반대 방향(R1, spec→code)은 더 단순하다: 명세에 FR-008을 새로 쓰면 → /sdd-sync의 check-fr-coverage가 "이 FR엔 검증 테스트가 없다"를 잡고 → "구현할까요?"를 물어 → TDD(RED→GREEN)로 코드를 만든다. 어느 방향이든: 기계가 어긋남을 찾고, 사람이 결정하고, LLM이 작성한다.

결정성 — 키·경계·PREFIX

Ownership 키의 결정성 — 누가 만들어도 같은 문자열

게이트는 "한 키 = 한 spec"을 비교한다. 그런데 그 키가 어떻게 만들어지는가는 규칙 없이 사람·LLM 판단에 맡겨져 있었다 — "같은 개념인데 표기가 달라" 거짓 green이 나오거나, "다른 개념인데 비슷해 보여" 차단이 누락됐다. 이 결정성 설계가 메우는 빈칸이다.

정규화 절대 규칙 (카테고리별)

카테고리정규화 규칙예시 결과
Entity스키마/모델 식별자 그대로 + trim·소문자. 단복수·대소문자 임의 변환 금지(스키마가 진실)recommendation, pjt_projects
SurfaceMETHOD path — 메서드 대문자, path 소문자, param {name} 표준형(:id·<id>→{id}), trailing slash 제거. 이벤트=event:name, job=job:namePOST /api/recommend/{id}
Capabilityentity.verb — 점 정확히 1개, 소문자, verb ∈ 허용집합(CRUD 기본 + config capabilityVerbs). 미등록 verb = 형식 위반recommendation.create, staff.recommend

키 생성 결정 절차 — 사람·LLM이 밟으면 같은 문자열

1 spec = 1 aggregate (분리의 결정적 기준)

"무엇이 하나의 spec인가"도 결정적 기준이 필요하다. "응집 묶음"은 주관적 — 대신 aggregate root(독립적으로 생성·삭제되는 핵심 Entity)를 경계 닻으로 쓴다.

소유

한 spec = 한 aggregate root를 Ownership.Entities에 소유. 그 aggregate를 변경하는 Capability·Surface도 함께.

참조

다른 aggregate는 Dependencies로 참조(dedup 제외). 예: recommendation spec이 staff를 읽을 때.

라우팅

새 FR → 그 FR이 변경하는 aggregate 식별 → 그 owner spec 개정. owner 없으면 새 spec(새 aggregate).

세 게이트가 한 닻으로 묶인다: dedup(1 Entity=1 spec = 경계 강제) · cohesion(Entities 2개 이상 = 여러 aggregate 삼킴 경고) · Dependencies(경계 밖 참조 명시). "독립 생성·삭제되는 root"라는 조작적 정의가 "응집 묶음"보다 훨씬 결정적이고, dedup이 사후에 강제한다.

spec PREFIX 거버넌스

접두어가 난립하면 게이트가 spec을 조용히 건너뛴다 — FEAT-001.md가 있어도 config에 FEAT가 없으면 FR이 통째로 누락된 채 거짓 green. 이를 막는 두 장치:

접두어의미기본 포함
SPEC기능 명세 (기본)✓
INFRA인프라 prerequisite spec✓
TEST테스트 전용 spec✓
그 외 (예: FEAT)표준 밖 — 사유 없이 추가 불가✗

조용한 누락 제거: 게이트가 specDir의 모든 ^[A-Z]+-\d{3}.*\.md를 스캔 — 접두어가 허용집합 밖이면 건너뛰지 않고 exit 1. 표준 밖 접두어는 specIdPrefixes에 등록 + prefixRationale에 사유(사유 없으면 exit 1). 기계는 사유의 존재만 강제하고 질은 리뷰가 본다.

$ node scripts/check-fr-coverage.mjs (FEAT-001.md 있고 FEAT 미등록 시 — 실측)

✗ PREFIX 위반:
  ✗ 미등록 접두어 "FEAT" (FEAT-001.md) — 표준 SPEC/INFRA/TEST. 임의 생성 금지, 필요하면 specIdPrefixes+prefixRationale에 사유와 함께 추가
채택 후 일상 운영

채택 후 궤도 한 바퀴 — hook·게이트가 언제·무엇을 하나

  2) spec 위치 = sdd/specs/ (docs/superpowers/specs/ 아님)
  3) PREFIX 표준 = SPEC / INFRA / TEST 만 (FEAT 등 임의 생성 금지)
