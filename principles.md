# 작업 원칙 (모든 SDD 작업 공통)

1. **도메인 범용** — 이 키트의 모든 템플릿/예시는 도메인 무관 placeholder. 적용 프로젝트의 도메인으로 치환해 쓴다. 특정 도메인(예: 티켓 시스템)을 다른 프로젝트에 강요하지 않는다.
2. **전부 정독** — 분석·리뷰 시 관련 파일을 빠짐없이 끝까지 읽는다. 부분 읽기·"안 읽음" 누락 금지.
3. **추측 금지·검증 우선** — "된다/구현됐다"고 말하기 전 **실제 실행·파일 확인**으로 증거를 댄다. 사실/판단을 `[검증]/[추론]/[미확인]`으로 구분. (`REALITY_CHECK.md`가 그 예)
4. **병렬 = 저비용 티어** — 워크플로우/fan-out 워커는 **빠르고 저렴한 모델 티어**를 기본으로(특정 모델명·벤더에 묶지 말 것 — "어떤 모델이든"). 종합·판정 등 정말 필요할 때만 상위 티어로 승격. 반복 중지/재시작으로 토큰 낭비 금지.
5. **병렬 실패 시 완료까지 재시도** — fan-out 에이전트가 실패하면 원인 진단 → 안전모드 재시도 → 최종 텍스트 폴백. 어떤 항목도 조용히 누락 금지.
6. **작성은 LLM, 승인은 사람** — spec 작성·갱신(top-down·bottom-up 모두)은 LLM이. 정본화 여부(bless/revert)는 사람이. 사람 승인 없는 코드→spec 자동 덮어쓰기 금지.
7. **언어 규약** — spec **정본은 영어**, 현지어본(예: 한국어)은 **생성만**(병행편집 금지=번역 드리프트 차단). 추적 닻은 **언어중립 FR-ID**. 기존 비영문 스펙은 일괄 변환 말고 **모듈 손댈 때 점진 영문화**. UI 언어는 spec 언어와 별개. (→ `SSOT.md` §6)
8. **폐기·삭제 수명주기** — 안 쓰는 spec은 **삭제 권장**(남기면 거짓 SSOT). 단 통제 제거: spec+코드+테스트를 **같은 PR로 원자적 삭제** → dangling `@covers`는 FR 게이트가 막음 → `MODULE_MAP`/Change Log에 사유 기록 → git이 히스토리 보존. **"spec 삭제 ⟹ 코드 삭제"는 방향은 맞지만 자동 아님**(task→구현으로 흐름). (→ `STRUCTURE.md` 폐기 수명주기)
9. **인프라 스펙은 배포 실제까지 검증 (CSP 무관)** — IaC 스펙은 spec↔코드뿐 아니라 **spec↔IaC↔running(실제 배포)** 까지 동기 확인. **어느 클라우드든 동일 원리** — AWS/GCP/Azure/온프렘, k8s든 서버리스든. git만 보지 말고 그 환경의 명령(`terraform plan`/`pulumi preview`/`cdk diff`·`aws/gcloud/az describe`·`kubectl get/diff`·Ansible `--check`)으로 drift 0인지 확인. 배포 자동화 미배선·CI 부재·콘솔 수동 변경이 흔한 drift 원인. 자동 sync는 외부 컨트롤러 소유 필드(오토스케일러 replicas 등)를 무시목록 제외. **관리형 DB도 동일(RDB·NoSQL 무관)** — 콘솔로 만든 테이블/인덱스/용량이 IaC 밖이면 drift. (→ `SSOT.md` §5b) **테스트 tier(로컬 vs 인프라):** 로컬 실행 가능성은 가정하지 말고 **능동적 probe**(인증/연결 시도)로 판정한다 — 로컬 강제(git 훅·TDD)는 인프라 의존 테스트를 실행하지 않고(`commands.test`=로컬 안전), 인프라 테스트는 자격증명·도달성 있는 개발서버·CI에서만(`commands.smoke`). probe 실패는 자원·사유와 함께 skip·회계에 명시(조용한 통과 금지). (→ METHODOLOGY "검증은 환경으로 계층화")
10. **다회 게이트-구동 수렴 (정확성 > 토큰) — 스펙 저술의 종결 조건** — 스펙을 새로 세우는 국면(최초 채택 adopt·재채택 readopt·마이그레이션 migrate)에서 저술의 종결은 "한 번 써서 게이트를 한 번 통과"가 아니라 **전 게이트를 돌려 위반이 0(선언된 리뷰 경계 advisory만 잔존)이 될 때까지 저술→게이트→수정을 반복**하는 것이다. 단일 패스로 끝내 부채(장식 bold·유령 capability·entity 없는 스펙·삼킨 aggregate)를 남기지 않는다 — 부채는 이후 라운드마다 재표면화돼 총비용이 더 크다. **토큰 절약보다 정합을 우선**한다(다회 실행이 정상 경로). 단, 도메인 판단 항목(어느 entity가 root인가·유령 명사가 실제 테이블인가)은 반복 대상이 아니라 사용자에게 묻는 대상이다(작성=LLM·확정=사람).
11. **언어·모델·인프라 무관 (범용의 핵심)** — 이 방법론은 **어떤 프로그래밍 언어·어떤 LLM/에이전트·어떤 인프라**에서도 같은 규율로 동작해야 한다. 구체화:
    - **언어 무관:** 게이트(`check-fr-coverage`·`check-ownership`)와 CI(`sdd-run`)는 코드에 언어 가정을 박지 않는다. 언어별 차이(테스트 파일 패턴·스캔 경로·lint/test 명령·중복 키 종류)는 **`sdd.config.json` 어댑터 한 장**으로만 표현(`tooling/sdd.config.presets.md`에 Python·Go·Rust·Java·Ruby·.NET·IaC 프리셋). 게이트는 텍스트 파서라(런타임 4판 중 하나로) 대상이 무슨 언어든 동작. `@covers` 주석 스타일도 무관(`//`·`#`·`--`).
    - **모델 무관:** 방법론에 특정 LLM·벤더 가정 없음. Spec Kit은 30+ 에이전트 비종속, 게이트는 모델과 독립적으로 CI에서 강제된다. Superpowers/슬래시명령은 *에디터 편의*일 뿐 — 핵심 규율(EARS·module>spec·FR↔test·작성LLM/승인사람)은 **어떤 에이전트로도 손으로 수행 가능**.
    - **인프라 무관(DB·CSP 포함):** SSOT 3계층의 "구조 SSOT"는 프로젝트마다 다르거나(미그레이션/스키마/proto/IaC) 없을 수 있다(순수 라이브러리·CLI). 인프라 의존이 없으면 `Infrastructure Prerequisites` 절·구조 계층을 생략한다.
      - **컴포넌트 무관 (어떤 미들웨어도 특정 금지):** DB(RDBMS·NoSQL)뿐 아니라 **캐시(Redis·Memcached…)·메시지브로커/스트림(Kafka·RabbitMQ·NATS·Pub/Sub·Kinesis…)·검색(Elasticsearch·OpenSearch…)·오브젝트스토리지·큐·함수** 무엇이든 방법론이 제품을 못 박지 않는다. **규칙: spec은 *역량/요구*를 적고 *제품*을 적지 않는다** — "내구성 있는 이벤트 로그가 필요" O / "Kafka 필요" X(제품은 그 프로젝트 결정, IP-XXX로 분리). 구조 SSOT 위치만 컴포넌트마다 다름(RDB=migration, NoSQL=인덱스/모델 정의, 스트림=토픽/스키마 레지스트리, 캐시=키 규약 등).
      - **CSP 무관:** AWS·GCP·Azure·온프렘 어디든 동일. drift 검증 *원리*는 같고 *명령*만 그 환경 것을 쓴다(§9). spec·게이트·config 어디에도 특정 클라우드 가정이 없다.
      - **CI/CD 도구 무관:** 게이트는 CLI라 **특정 CI/CD 도구도, git 호스트도 요구하지 않는다.** 로컬·`make`·git pre-push 훅·사내든 클라우드든 **어떤 CI/CD 도구**에서 호출해도 된다. 샘플 워크플로우·도구별 동일 명령은 `tooling/ci-examples.md`. **Spec Kit(spec 작성) ≠ CI/CD 도구(게이트 실행)** — Spec Kit은 git 호스트·CI와 무관하게 동작한다. SSOT를 강제하는 건 *매 변경마다 게이트가 자동 실행돼 red면 막는다*는 사실이지 특정 CI/CD 제품이 아니다.
      - **런타임 무관(게이트 자체):** 게이트는 텍스트 파서라 특정 *언어 런타임*에 묶지 않는다. **솔직한 사실: 프로그램은 *무언가*가 실행해야 하므로 "0 의존"은 불가능하다.** 핵심은 그 의존이 *프로젝트 언어가 아닐 것*. 동작 동일한 4판 동봉(전부 실행 검증됨):
        - **`go-gate` (Go → 단일 정적 바이너리) — 사실상 모든 언어 커버, 권장 배포물.** `CGO_ENABLED=0`이면 libc도 안 묶인 자립 바이너리(검증: linux/amd64 `statically linked`). 한 소스에서 linux·darwin·windows × amd64·arm64 크로스컴파일(검증). **소비자는 Go조차 필요 없이 바이너리만 받아 실행 → 어떤 언어(Go·C·Rust·COBOL·…)든 인터프리터 0.** 네이티브 Windows도 커버. 릴리스 워크플로우 `sdd-gate-release.yml`가 전 플랫폼 빌드.
        - **`sdd_gates.sh` (POSIX 셸)** — 빌드/다운로드 없이 소스 그대로 돌릴 때. `sh`+`grep`+`awk`+`jq`만(유닉스/CI 기본). 실증: Go 픽스처를 `go` 미설치로 게이트 통과.
        - **`sdd_gates.py` (Python)** · **`*.mjs` (Node)** — 그 스택에 인터프리터가 이미 있을 때의 편의.
        - 우선순위: **배포 파이프라인이 있으면 Go 바이너리**(가장 보편), 빌드 없이 즉시 돌릴 땐 셸판, 해당 스택이면 Python/Node판. (알고리즘 명세: `SSOT.md` §4·`DEDUP.md`)
        - **확정(2026-06-29): 게이트 런타임은 이 4판으로 동결 — Go·Node·Python·셸.** Rust·C++ 별도 구현은 **안 한다**(Go 정적 바이너리가 그 컴파일-바이너리 역할을 이미 커버하므로 중복). 새 언어 *프로젝트*는 게이트를 새로 만들 필요 없이 `sdd.config.json`만 맞추면 위 4판 중 하나로 그대로 동작한다.
      (→ `STRUCTURE.md` §SSOT 3계층, `SSOT.md` §5b, `tooling/sdd.config.presets.md`)
