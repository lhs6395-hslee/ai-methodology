소유권 유일성 게이트
— 스펙이 늘어도 중복이 안 쌓이게

스펙 간 중복을 “사람이 판단”에서 “CI가 강제”로 바꾸는 규칙 — 무엇을·왜·어떻게 쓰는지 · 2026-06-29

30초 요약
문제
기능이 늘면 비슷한 스펙이 여러 개 생기고, 같은 내용을 두 곳이 “진실”이라고 주장한다(중복 → drift).
해법
각 스펙이 자기가 소유하는 키를 선언한다. 한 키는 한 스펙만 소유 — 두 스펙이 같은 키를 가지면 CI가 exit 1로 막는다.
당신이 할 일
스펙 쓸 때 ## Ownership 블록 채우기 · 새 요구가 오면 “이 키 주인이 이미 있나?” 먼저 조회하기.

01왜 필요한가 — 막지 않으면 생기는 일

스펙 기반 개발(SDD)에서 스펙은 “그 기능의 단일 진실원(SSOT)”이어야 한다. 그런데 기능이 쌓이면 자연스럽게 이런 일이 벌어진다:

중복이 쌓이는 전형적 시나리오

지난달 SPEC-014 프로젝트 인력배정을 썼다. 이번에 “외부 협력사 담당자도 배정” 요구가 온다. 새 스펙 SPEC-022 협력사 배정을 만든다. 둘 다 같은 테이블(pjt_project_staff)·같은 동작(staff.assign)을 건드린다. 이제 “배정 규칙의 진실”이 두 곳에 흩어진다. 한쪽만 고치면 둘이 어긋나고(drift), 코드는 어느 스펙을 따라야 할지 모른다.

기존 방법론은 “이게 중복인가?”를 사람·LLM 판단에 맡겼다. 그런데 리뷰어는 14번 스펙의 존재를 잊고, LLM은 누락을 낸다. 판단에 맡기는 한 중복은 새어 들어온다. 그래서 이 키트는 중복 판정을 기계가 강제하도록 바꿨다.

02중복은 두 종류 — 잡는 방법이 다르다

“중복”을 한 덩어리로 보면 자동화가 안 된다. 두 종류로 나누면 각각 맞는 도구가 보인다.

계층 1 · 구조적 중복 (게이트가 강제)

두 스펙이 같은 대상(같은 테이블/route/동작)을 소유. 집합 조회로 명확히 판정되므로 CI가 자동으로 막는다. 이게 1차 방어선.

계층 2 · 의미적 중복 (리뷰가 보조)

대상은 다른데 말만 바꾼 같은 요구(reworded). 게이트는 못 잡는다 → 같은 영역 이웃 스펙끼리 좁혀서 사람이 리뷰. 확률적이라 강제 못 함.

구분구조적 중복의미적 중복
무엇두 스펙이 같은 키 소유키는 다른데 의도가 같음
예두 스펙 다 staff.assign 소유“인력 배정” vs “리소스 할당” (사실 같은 것)
판정집합 조회 — 결정적유사도 추정 — 확률적
도구check-ownership.mjs이웃 리뷰 + (선택) 임베딩 유사도
강제력CI 게이트 (exit 1)좁힌 리뷰 보조

핵심 원칙: LLM·사람 리뷰는 누락을 낸다 → 결정적 게이트가 1차, 확률적 리뷰는 2차.

03핵심 개념 — 스펙이 “소유하는 키” 3종

게이트의 전부는 이 한 문장이다: “하나의 키는 정확히 한 스펙만 소유한다.” 그럼 키가 뭔가? 스펙이 권위를 갖는 대상을 세 종류로 본다.

Entity

도메인 객체 · 테이블

이 스펙이 정의·소유하는 데이터. 보통 DB 테이블이나 핵심 도메인 객체.

pjt_projects
pjt_project_staff
Surface

route · 화면 · job

외부에 노출되는 표면. API 엔드포인트, 화면 경로, 배치/이벤트 핸들러.

POST /api/pjt/recommend
/tools/pjt-management/new
Capability

Entity × Action

“무엇을 할 수 있는가”. 객체와 동작의 조합으로 한 능력을 가리킨다.

project.create
staff.assign

스펙마다 자기가 소유하는 키를 ## Ownership 블록에 적는다. 이 블록이 게이트가 읽는 “소유권 등기부”다.

## Ownership   ← 모든 스펙이 선언
- **Entities**: pjt_projects, pjt_project_staff
- **Surfaces**: POST /api/pjt/recommend, /tools/pjt-management/new
- **Capabilities**: project.create, staff.assign

미선언 스펙은 일단 경고만(점진 도입). 팀이 자리잡으면 --strict로 “모든 스펙이 Ownership 선언 필수”로 승격한다.

04실무 — 새 요구가 오면 (결정 트리)

새 기능 요구를 받으면 새 스펙을 만들기 전에 이 순서로 판단한다. 핵심은 “새로 만들까?”가 아니라 “이 키 주인이 이미 있나?”를 먼저 묻는 것.

  1. 키를 뽑는다. 이 요구는 어떤 Entity를 건드리나 / 어떤 Surface인가 / 어떤 Capability인가?
  2. 그 키의 주인을 조회한다. MODULE_MAP 등기부 또는 check-ownership으로 이미 소유한 스펙이 있는지 본다.
  3. 있으면 → 그 스펙을 개정한다 (새 스펙 금지). 같은 범위면 owner 스펙에 FR을 추가. 완전히 새 범위일 때만 새 스펙을 만들고 키를 등록한다.
올바른 흐름

“외부 협력사 담당자 배정” 요구 → 키 산출: staff.assign. 조회해보니 SPEC-014가 이미 소유. → SPEC-014에 FR을 추가해 개정. 진실이 한 곳에 유지된다.

게이트가 막는 실수

같은 요구로 새 SPEC-022를 만들고 거기에도 staff.assign을 선언. → 두 스펙이 같은 키 소유 → CI가 exit 1로 차단. 병합 전에 강제로 멈춘다.

05CI에서 막히면 — 이렇게 읽고 고친다

게이트가 충돌을 잡으면 어느 키를 어느 스펙들이 다투는지 그대로 출력한다. 추측할 필요가 없다.

$ sdd-gate ownership        # 또는: sh sdd_gates.sh ownership · python sdd_gates.py ownership · node scripts/check-ownership.mjs
Ownership 게이트: spec 3개 중 2개가 Ownership 선언.
✗ 중복 소유(구조적 중복) 1건:
  [Entities] "pjt_projects" ← SPEC-001 + SPEC-002  → 한 spec으로 통합/개정 필요
# exit 1 — 빌드 중단

게이트는 언어·런타임 무관 4판(Go 바이너리·셸·Python·Node, 동작 동일) — 가진 쪽으로 돌린다. 키 종류(Entities 등)는 sdd.config.json의 ownershipCategories로, 비-웹은 Modules·Symbols·Artifacts 등으로 바꾼다. CI/CD 도구도 무관(특정 도구 한정 아님).

고치는 법 (3단계)

  1. 진짜 주인을 정한다. pjt_projects의 권위를 가질 스펙 하나를 고른다(보통 먼저 만든·더 핵심인 쪽).
  2. 다른 스펙에서 그 키를 뺀다. 그 스펙이 정말 그 데이터가 필요하면, 소유하지 말고 owner 스펙을 참조하거나 owner 쪽으로 FR을 옮긴다.
  3. 다시 돌린다. exit 0 — 모든 키가 유일이 나오면 통과. MODULE_MAP 등기부도 정리한다.
해결 후

✓ 구조적 중복 없음 — 모든 Entity/Surface/Capability 키가 유일.  exit 0

06이 게이트는 어디서 왔나 — 논문과의 관계

먼저 짚고 갈 것

이 게이트는 어떤 논문의 알고리즘을 옮긴 게 아니다. 소프트웨어공학 1차 원칙에서 직접 설계했다 — ① DDD bounded context(한 능력은 한 곳이 소유) ② Single Source of Truth(진실은 한 곳) ③ 집합 유일성(한 키 = 한 owner). 이게 약점이 아니라 강점이다: 1차 원칙 기반이라 판정이 결정적이고, 그래서 CI에서 빌드를 깰 수 있다(확률적 모델이면 못 한다).

아래 논문은 다른 문제 — 의미적(NLP) 중복 탐지를 다룬다. 우리 게이트의 출처가 아니라, 계층 2(의미적 리뷰)의 근거이자 참고 구현으로만 인용한다.

Transfer learning for conflict and duplicate detection in software requirement pairs

Garima Malik, Savas Yildirim, Mucahit Cevik, Ayse Bener, Devang Parikh · arXiv:2301.03709 (cs.SE) · 2023 제출, 2024 개정

문제
충돌·중복 요구사항을 자동 식별해 요구 표현의 일관성을 높인다.
방법
요구문 “쌍(pair)” 분류로 형식화 → SR-BERT(Sentence-BERT+Bi-encoder)를 다단계 파인튜닝(transfer learning).
결과
데이터셋 4종 교차평가에서 SR-BERT가 큰 데이터셋에서 최고 성능 — 문장쌍 NLP가 중복/충돌 탐지를 효과적으로 자동화.
왜 인용
(a) “말이 다른 같은 요구”가 실재한다는 근거, (b) 계층 2를 만들 때의 참고 구현. 단 확률적이라 CI 강제용으론 부적합 → 1차는 결정적 게이트여야 한다.

중복을 구조적(같은 키)과 의미적(말 바꾼 같은 요구)으로 나눠, 구조적은 게이트로 강제(1차), 의미적은 좁힌 리뷰로 보조(2차). 덕분에 “중복인가?”가 사람의 기억력이 아니라 CI가 막는 기계적 사실이 된다.
